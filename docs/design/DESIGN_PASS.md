# ============ FACETS (condensed) ============

## [architecture] Application architecture, process model & tech stack
SUMMARY: Build "sabiorg" as an Electron 33+ desktop shell wrapping a pure-TypeScript engine library, because the heavy work (fpcalc, ffprobe, image perceptual hashing, MusicBrainz HTTP, SQLite writes over 1M rows) is overwhelmingly Node-native and async-IO-bound — exactly where Electron's first-class Node integration beats a Tauri-v2 + Node-sidecar setup that would reintroduce a second runtime, a fragile IPC seam, and packaging of a bundled Node binary anyway. The architecture is three-tier: a thin Electron renderer (React + TS, zero Node access), a coordinator main process, and a separate long-lived utilityProcess "engine" host that owns the DB and drives a bounded worker_threads pool plus child_process subprocess pools for native binaries. State lives in a single embedded better-sqlite3 database (WAL mode, FTS5 for search) accessed only from the engine process, with all UI reads/writes flowing through a typed, backpressure-aware IPC contract. The same engine package is consumed verbatim by a headless CLI, so scan/reconstruct/enrich logic is never duplicated or coupled to Electron.
RECO: Choose Electron over Tauri-v2-with-TS-sidecar. The "modern TypeScript" mandate plus 1M-item native workloads makes Tauri's Rust core a liability you pay for without using: you'd still ship a Node sidecar for the engine, so you get Tauri's packaging/IPC complexity AND a full Node runtime — the worst of both. Electron gives you one runtime, native better-sqlite3/sharp/worker_threads, mature auto-update, and a single language across UI, main, engine, and CLI. Mitigate Electron's bundle size and renderer-security reputation with strict process isolation (contextIsolation on, nodeIntegration off, sandbox on) and by running the engine in a utilityProcess, not the renderer.
DECISIONS:
  - Desktop shell: Electron vs Tauri v2 + Node sidecar => Electron 33+
  - Where the engine runs => Separate Electron utilityProcess (own PID), NOT in renderer and NOT only in worker_threads of main
  - Database engine for 1M+ rows on desktop => better-sqlite3 (SQLite) with WAL, FTS5 for search; sqlite-vec extension only if/when embedding-based similarity is needed
  - Worker model for CPU-bound tasks => Piscina-managed worker_threads pool, pinned to physical core count, with batched streaming tasks
  - Native binary execution (fpcalc, ffprobe/ffmpeg) => execa-driven child_process with a per-binary p-queue concurrency cap, binaries bundled per-platform
  - IPC design => Typed contract package with zod-validated messages; electron-trpc (or a thin tRPC-over-IPC link) for renderer<->main request/response, plus an explicit event/stream channel for progress with credit-based backpressure
  - Monorepo & package structure => pnpm workspaces + Turborepo: packages/engine, packages/ipc, packages/cli, apps/desktop (electron-vite), plus packages/native-binaries
  - Config & secrets (API keys for AcoustID/Discogs/MusicBrainz) => Non-secret config in a versioned JSON/TOML under app.getPath('userData'); API keys encrypted at rest via Electron safeStorage (OS keychain-backed), never bundled in the build
  - Build, packaging & auto-update => electron-vite for dev/build, electron-builder for packaging + code signing, electron-updater with a self-hosted/static update feed (e.g. S3/GitHub Releases)
  - Logging & telemetry => Local-first structured logging with pino to rotating files under userData/logs (one stream per process: main, engine, workers), surfaced in an in-app log viewer; OpenTelemetry optional and OFF by default
  - Language/runtime baseline & test stack => Node 22 LTS (matches local v22.16.0), TypeScript 5.6+ strict, Vitest for unit/integration, Playwright for Electron E2E
ALGORITHMS:
  - Credit-based IPC backpressure for progress streaming: Engine emits scan/enrich progress in batches over a dedicated MessagePort channel. Renderer holds N credits; each batch consumes one and the renderer returns a credit after rendering. Engine pauses emission at zero credits. Prevents unbounded buffering when the UI can't keep up with a 1M-item firehose. Thresholds: start with N=8 batches of ~250 items; tune via observed render latency.
  - Bounded job scheduler with resumable checkpoints: Single scheduler in the engine process pulls work from a SQLite-backed queue (states: pending/running/done/failed). Stages: walk -> probe/tag -> fingerprint -> enrich -> reconstruct -> organize. Each file's per-stage state is checkpointed transactionally; concurrency caps per stage (e.g. fingerprint=cores, enrich=network-limited). On startup, requeue any 'running' rows. Confidence is not relevant here; correctness comes from idempotency keyed by file content hash + path.
  - Batched worker dispatch with memory bounding: Scanner streams file paths and groups them into fixed-size batches (default 200) sent to Piscina. Workers parse tags/compute perceptual hashes and return compact records (no raw buffers cross the boundary). maxQueue caps in-flight batches to (cores*2); scanner awaits queue drain before producing more, giving natural backpressure from disk -> workers -> DB writer.
LIBS: electron (33+), electron-vite + vite 6, electron-builder + electron-updater, better-sqlite3, drizzle-orm, sqlite-vec, piscina, execa, music-metadata, sharp, ffprobe-static / ffmpeg-static (or vendored builds), chromaprint/fpcalc (vendored), undici, electron-trpc + @trpc/server, zod, pino + pino-roll, piscina-friendly batching utils (p-queue, p-limit), tsup, turborepo + pnpm, vitest + @playwright/test
RISKS:
  - [high] Native-module ABI breakage (better-sqlite3, sharp) across Electron upgrades and the 3 platforms causes runtime crashes or broken installers -> Pin Electron and native deps; use prebuilt binaries where available and electron-rebuild in CI; run a packaged-app smoke test (launch + open DB + read a tag + run fpcalc) on Windows/macOS/Linux in CI before every release.
  - [high] Memory blowup on 1M-item scans (holding rows, buffers, or unbounded IPC/worker queues in memory) -> Stream everything: cursor-based DB reads, batched worker tasks with maxQueue, credit-based IPC backpressure so the renderer paces the engine, and never materialize full result sets — virtualize the UI and paginate queries. Add a heap watchdog in the engine process.
  - [medium] External service rate limits / outages (MusicBrainz 1 req/s, AcoustID quotas, Discogs throttling) stall enrichment or get the app banned -> Token-bucket limiters per service with correct User-Agent, persistent on-disk response cache, exponential backoff, resumable enrichment jobs, and the locked offline-fallback path to local tags + heuristics so scans never block on the network.
  - [medium] Long scan interrupted by crash/quit/power loss leaves the DB or the copy-organize step in a half-done state -> Resumable job model with per-file state checkpoints in SQLite; idempotent steps keyed by content hash; copy-then-atomic-rename into the destination tree; WAL + periodic checkpoints; on startup, detect and resume in-flight jobs.
  - [medium] Renderer security regression exposes Node/fs to web content -> Enforce contextIsolation:true, nodeIntegration:false, sandbox:true; minimal preload surface; strict CSP; no remote content loaded into privileged windows; all fs/DB access only via validated IPC to the engine.
  - [low] Electron bundle size and update payloads are large for a media tool users may install on modest machines -> Use electron-updater delta updates, prune locales and unused Chromium features, externalize bundled media binaries so they update independently, and asar-pack the app.
  - [medium] ffmpeg/ffprobe redistribution licensing (GPL components) complicates shipping -> Ship LGPL-configured builds or document the GPL obligations; isolate ffmpeg behind the subprocess adapter so it can be swapped or made an optional user-supplied binary if licensing requires.
  - [low] Album reconstruction logic accidentally coupled to Electron, breaking CLI reuse and testability -> Hard rule: @sabiorg/engine has zero electron imports, enforced by an ESLint import boundary / dependency-cruiser check in CI; the CLI package is a first-class consumer exercised in tests.
MILESTONES:
  - M0 — Skeleton & boundaries: pnpm+Turbo monorepo (apps/desktop, packages/engine|ipc|cli|native-binaries); Electron 33 + electron-vite shell with utilityProcess engine host; typed IPC contract (zod) and a hello-world tRPC call; ESLint import-boundary check enforcing zero electron imports in engine; CI matrix builds a packaged smoke app on Win/mac/Linux.
  - M1 — Scan & index (music only): Streaming filesystem walker; Piscina worker pool reading tags via music-metadata; better-sqlite3 schema + Drizzle migrations; resumable job scheduler with checkpoints; credit-based progress streaming to a virtualized renderer list; bounded-memory verified on a synthetic 1M-file fixture.
  - M2 — Reconstruct & metadata edit: Album reconstruction grouping over tags+paths; review/approve UI; metadata editor writing 'proposed' rows (originals untouched); copy-organizer that writes the new Artist/Album/NN-Track tree via temp-file+atomic-rename.
  - M3 — Enrichment (hybrid + offline fallback): Subprocess pool for fpcalc; AcoustID lookup; MusicBrainz/Discogs/Cover Art Archive clients via undici with per-service token buckets and on-disk enrichment_cache; offline-fallback to local tags+heuristics; secrets via safeStorage.
  - M4 — Insights: Collection insights (genre/format/lossless/bitrate/dup/completeness) and owner-profiling (classic-vs-new %, acquisition cadence from timestamps vs release years) via summary tables / optional DuckDB attach; dashboard charts.
  - M5 — Images & video + packaging: sharp-based image hashing/grouping; ffprobe/ffmpeg video probing and frame perceptual hashing; electron-builder signed installers + electron-updater delta updates; in-app pino log viewer; full E2E via Playwright.

## [ingestion] Scanning & ingestion pipeline at scale (100k–1M+ files)
SUMMARY: A staged, cheap-first ingestion engine that decouples filesystem discovery from progressively more expensive enrichment so we never fingerprint or hit the network for files that haven't changed. Discovery is a fast async walk (fdir) feeding a SQLite-backed work queue; each file flows stat → probe/tag-read → content-hash (xxhash gate) → audio-fingerprint (fpcalc) → online-enrich, with each transition guarded by an mtime/size/inode change-detection cache so re-scans touch only deltas. The engine runs in a Node sidecar/worker pool with bounded per-stage concurrency, persistent checkpointing for crash recovery, and a throttled progress event stream to the UI. Target: full cold walk + tag-read of 1M files in single-digit minutes; warm re-scans in seconds because 99% of files short-circuit at the stat stage.
RECO: Make SQLite (better-sqlite3, WAL mode) the durable backbone of the pipeline — not just final storage but the work queue itself. Every file is a row with a per-stage status column; "the queue" is a query (`WHERE stage_state='pending' ORDER BY cost_tier`). This gives you free crash recovery, resumability, idempotency, and incremental re-scan for nearly zero extra code, and it is the single decision that makes 1M-item scale tractable on a desktop without inventing a bespoke persistence layer.
DECISIONS:
  - What drives durable work-queue + resumability => SQLite catalog with per-stage status columns acts AS the queue (better-sqlite3, WAL)
  - Filesystem traversal library => fdir async builder API
  - Stage ordering / cost tiers => stat → probe+tag-read → content-hash → audio-fingerprint → online-enrich, each gated by change-detection and 'will-this-ever-be-needed' checks
  - Content hash algorithm => xxhash3 (@node-rs/xxhash) for the byte-dedup gate; keep it SEPARATE from the audio fingerprint
  - Concurrency model inside the desktop app => Node sidecar/main-process engine with a piscina worker pool for CPU work + child_process for fpcalc; per-stage bounded concurrency
  - Tag reading approach => music-metadata v11 streaming for the read path; node-taglib-sharp reserved for the WRITE path to the new tree
  - Change detection key => (size, mtimeMs, dev+ino) tuple, content-hash only on tuple mismatch
  - Symlink / junction policy => Do NOT follow symlinks/junctions by default; maintain a realpath+inode visited-set; treat reparse points on Windows as opt-in
ALGORITHMS:
  - Cheap-first staged pipeline with cost-tier gating: Model each file as a state machine across stages S0..S4: S0 discover(path), S1 stat(size,mtime,ino), S2 probe+tag(codec/bitrate/duration/tags), S3 hash(xxhash3), S4 fingerprint(fpcalc), S5 enrich(online). Scheduler selects work `WHERE next_stage=N AND state='pending' ORDER BY tier LIMIT batch`. A stage runs ONLY if (a) change-detection invalidated its cached result, AND (b) a need-predicate is true. Need-predicates prune aggressively: hash only if dedup enabled or file changed; fingerprint only if audio AND (tags missing/ambiguous OR album reconstruction confidence below threshold); enrich only if fingerprint or strong-tag key exists and network is available. Each completed stage commits a row patch in a transaction (checkpoint).
  - Incremental / resumable change detection: On (re)scan, for each discovered path look up catalog row by (dev,ino) then fallback by path. If row exists and (size,mtimeMs) match cache → mark all stages 'valid', short-circuit (file fully skipped, the common case → seconds-scale warm scans). If tuple mismatches → reset S2..S5 to 'pending', keep S1. If found by ino but path differs → it moved: update path, preserve hash/fingerprint/enrichment (no re-work). New path with no row → full pipeline. Paths in catalog not seen this epoch → mark candidate-deleted (soft, pending user confirm).
  - Crash recovery / checkpoint protocol: Every stage transition is `BEGIN; UPDATE file SET state=...,result=... WHERE id=?; COMMIT`. Before dispatching to a worker, mark row state='in_flight' with a worker/epoch stamp. On engine startup, run `UPDATE file SET state='pending' WHERE state='in_flight'` (reclaim orphaned work) — completed stages are durable and never repeated. A scan_epoch counter groups a run so progress totals survive restart. Periodic `PRAGMA wal_checkpoint(TRUNCATE)` bounds WAL growth.
  - Bounded-concurrency, backpressured stage scheduling: Each stage owns a p-queue/piscina with an explicit concurrency cap tuned to its bottleneck: discovery 1 (sequential walk), stat high (I/O-light), probe medium, hash = ~#cores (CPU+disk), fingerprint = min(#cores, small) (CPU heavy), enrich low + interval-rate-limited (network/API ToS). Backpressure: scheduler only pulls a new batch for stage N when its queue depth < cap*2, so no stage can buffer 1M tasks in memory. Pause = stop pulling + AbortController.abort() in-flight; resume = re-query pending.
  - Bounded-memory streaming hash & probe: Never read whole files. Hash: createReadStream with fixed highWaterMark (e.g. 1MB), feed chunks to xxhash3 incremental, discard. Probe: music-metadata.parseStream reading only header/tag region (bounded). Discovery batches paths (e.g. 5k) into SQLite inserts then releases. Hold no global arrays of file objects — everything flows through SQLite.
  - Exact-duplicate detection (xxhash) vs same-recording detection (fingerprint): Two-level dedup: (1) byte-identical via xxhash3 — group rows sharing content_xxhash AND size; cheap, exact, found at S3. (2) same-recording-different-encode via Chromaprint/AcoustID at S4/S5 — only run on the deduped representative set, not every file. Keep them as distinct columns/relations so insights can report 'N exact dupes reclaiming X GB' separately from 'M alternate encodings of same track'.
LIBS: fdir, better-sqlite3, music-metadata (v11), @node-rs/xxhash, fpcalc (Chromaprint) + fpcalc npm wrapper, piscina, p-queue, ffprobe (ffmpeg) via child_process, node-taglib-sharp
RISKS:
  - [high] Native-dependency packaging hell: better-sqlite3, @node-rs/xxhash, and the fpcalc/ffprobe binaries must build/bundle correctly across Win/macOS/Linux and arch (x64/arm64), and Electron needs ABI rebuilds. -> Pin prebuilt-binary versions; use electron-rebuild or Tauri sidecar packaging with per-target binaries in CI matrix; offer hash-wasm as a pure-WASM fallback; ship fpcalc/ffprobe as bundled resources with a startup capability check + graceful offline degrade if absent.
  - [high] fpcalc/online enrichment is slow and rate-limited (MusicBrainz ~1 req/s, AcoustID quotas) — naively fingerprinting/enriching 1M files takes days and risks API bans. -> Hard-gate S4/S5 behind need-predicates so only unidentified/ambiguous files reach them; interval-rate-limit the enrich queue; aggressive local cache of fingerprint→MBID; batch lookups where the API allows; let scan complete usefully offline and enrich lazily in background.
  - [medium] Network drives / coarse-granularity or no-inode filesystems (SMB, exFAT) break change detection and tank traversal throughput. -> Detect filesystem/drive type at root; fall back to (size+path+content-hash-on-suspicion) when inode/mtime unreliable; lower concurrency and warn the user on network roots; make hash-on-suspicion the safety net.
  - [medium] SQLite write contention / WAL growth under high-throughput bulk insert from many workers can stall or bloat. -> Single writer pattern (workers return patches, one thread commits) or serialized write queue; batch in transactions of N rows; WAL mode + busy_timeout; periodic wal_checkpoint(TRUNCATE); prepared statements.
  - [medium] Symlink/junction cycles or out-of-root escapes cause infinite walks, double counting, or reading unintended files. -> Don't follow links by default; realpath+inode visited-set cycle guard; clamp traversal to the configured root; treat Windows reparse points as opt-in.
  - [low] UI flooded by per-file progress events at 1M scale, causing IPC backpressure and UI jank. -> Coalesce/throttle progress to ~4–10 Hz with aggregate counters; push deltas not full state; keep heavy data in SQLite and let UI query on demand.
  - [medium] Memory blowup from accumulating file objects/promises in flight. -> SQLite as the only large collection; bounded queues with depth caps; streaming reads; never Promise.all over the full set; cap concurrent open fds.
MILESTONES:
  - M1 — Walk + Catalog (MVP): fdir discovery → batched insert into better-sqlite3 catalog (path, size, mtime, dev+ino); ignore rules + extension allowlist; symlink cycle guard; basic throttled progress to UI. Goal: cold-walk 1M files into the DB in single-digit minutes, re-walk in seconds.
  - M2 — Probe + change-detection + resumability: S1 stat + S2 music-metadata streaming tag/codec read; (size,mtime,ino) change-detection gate; per-stage state-machine columns; crash recovery (in_flight→pending on startup); scan_epoch + deleted detection. Goal: true incremental re-scans touching only deltas.
  - M3 — Hash + dedup + worker pool: piscina worker pool; streamed xxhash3 hashing gated by change-detection; exact-duplicate grouping + 'reclaimable GB' insight; per-stage bounded concurrency + backpressure + pause/resume via AbortController. Goal: full hash pass within disk-bandwidth limits, responsive UI.
  - M4 — Fingerprint + online enrich (gated, offline-graceful): fpcalc/Chromaprint stage gated by need-predicates; fingerprint cache; rate-limited enrich queue (AcoustID/MusicBrainz) with local caching and offline fallback to tags/heuristics; same-recording dedup. Goal: only ambiguous files fingerprinted/enriched, no API bans, scan useful fully offline.
  - M5 — Hardening & adaptivity: Network-drive/coarse-FS detection + fallbacks; adaptive concurrency tuning; WAL checkpoint management; observability counters per stage (throughput, ETA, error rates); fuzz/large-corpus test harness. Goal: production-grade at 1M+ across platforms.

## [datamodel] Unified media data model & storage schema
SUMMARY: A single normalized relational schema (SQLite via better-sqlite3, WAL) unifies music/image/video around a polymorphic MediaFile spine with media-type satellite tables (TrackFile/PhotoFile/VideoFile) and a MusicBrainz-aligned logical layer (Artist -> ReleaseGroup -> Release -> Medium -> Recording -> Track). Every derived field is stored as a provenance-tracked MetadataClaim (field, value, source, confidence, observedAt) rather than a bare column, and a materialized ResolvedField projection per entity gives fast reads while the UI can always show "from tag" vs "from MusicBrainz" vs "inferred". Human-in-the-loop groupings live in a candidate -> reviewed -> confirmed state machine (AlbumCandidate/Grouping/GroupingDecision) so proposed reconstructions never overwrite confirmed ones. SQLite delivers ACID, FTS5 full-text, and sqlite-vec similarity fully in-process, fitting a bounded-memory, resumable, offline-capable desktop engine far better than an embedded server or document DB.
RECO: Use SQLite (better-sqlite3, synchronous, WAL) as the single embedded store, Drizzle ORM for typed schema + forward-only migrations, FTS5 for full-text, and sqlite-vec for fingerprint/perceptual-hash similarity. Model every derived attribute as an append-only MetadataClaim row (entity, field, value, source, confidence) and materialize one ResolvedField row per (entity, field); this single decision makes provenance, offline fallback, re-enrichment, and human overrides all fall out naturally instead of being bolted on.
DECISIONS:
  - Embedded store engine => SQLite via better-sqlite3, not a document DB (PouchDB/RxDB) and not PGlite/Postgres
  - How derived metadata is stored => Append-only MetadataClaim (EAV-with-provenance) + materialized ResolvedField projection, NOT bare columns
  - Unification vs format-specific detail => Polymorphic MediaFile spine + per-type satellites (TrackFile/PhotoFile/VideoFile); shared Asset/Grouping abstraction over Release/PhotoEvent/Series
  - MusicBrainz alignment depth => Adopt the MB hierarchy (Artist, ArtistCredit, ReleaseGroup, Release, Medium, Recording, Track) as first-class tables with ExternalId to MBID/Discogs/AcoustID
  - Proposed vs confirmed grouping coexistence => Separate Grouping (confirmed) from AlbumCandidate (proposed) with a status FSM and an immutable GroupingDecision log; membership rows carry source=auto|human
  - Source-of-truth for organized output => OrganizePlan/OrganizeAction journal with content-hash idempotency keys; originals read-only, every action a COPY recorded with src+dst+hash+status
ALGORITHMS:
  - Album reconstruction -> candidate generation: Blocking then scoring. (1) Block files into possible-album buckets by signals: same parentDir, normalized resolved album tag, AcoustID/Recording cluster, shared embeddedCoverHash, btime proximity. (2) Score cohesion: tag agreement on album+albumArtist+year, contiguous/complete trackNo sequence, consistent codec/source, MB Release match via ExternalId. (3) If score>threshold create AlbumCandidate(status=candidate) + CandidateMember with per-member signalsJson + confidence. (4) If a confirmed Grouping already covers the files, emit as superseded/no-op unless it materially improves (then status=candidate, supersedes link). Completeness vs Release.expectedTrackCount when MB-matched, else vs max observed trackNo/disc. Never auto-confirm; flag high-confidence as auto-confirmable for one-click review.
  - Duplicate detection (exact + near): Exact: group MediaFile by contentHash (blake3/xxh3) -> DuplicateGroup(exact_bytes); pre-filter with quickHash. Near-audio: block by AcoustID id when present, else sqlite-vec nearest-neighbor on fp vectors under a distance threshold; same Recording across encodings -> same_recording. Near-image/video: pHash Hamming distance under threshold via BK-tree blocking then exact Hamming; sampled keyframe hashes for video. Keeper chosen by quality heuristic: lossless > higher bitrate > resolution > size > most-complete tags.
  - Incremental/resumable scan reconciliation: Per path compute quickHash; if MediaFile.path exists and (mtime,size) unchanged -> bump lastSeenAt, skip. If changed -> recompute contentHash + reprobe, append new claims (supersede old). New path -> insert MediaFile(discovered). After walk, MediaFile with lastSeenAt<run.startedAt under a scanned root -> set removedAt (soft delete). ScanCursor checkpoints per shard/phase for crash resume; writer worker commits batches transactionally.
  - Provenance resolution (claims -> ResolvedField): For each (entityType,entityId,field), select winning claim by MergePolicy: human-override always wins; else max source priority, tie-broken by confidence then recency. Write/refresh ResolvedField pointing at resolvedClaimId with humanOverride flag. Recompute incrementally on new claim insert or user edit; pure + idempotent.
  - Owner profiling from build history: Acquisition cadence: histogram btime (fallback ctime) by month/year -> bursts vs steady. Classic-vs-new: for files with resolved/MB release year compute (acquisitionYear - releaseYear), bucket classic vs new, aggregate % and per-decade taste. Taste: weighted genre/style distribution from ResolvedField genre. Persist OwnerProfileSnapshot per ScanRun for trends.
  - Organize copy execution (idempotent journal): From confirmed Groupings render dstRelPath via templateJson using ResolvedField values (FS-sanitized, collision disambiguators). Insert OrganizeAction(pending) with idempotencyKey. Execute: if dst exists and dstHashAfter==sourceHash -> skipped; else copy bytes (verify hash), write_tags/write_cover from resolvedTagsJson, set dstHashAfter, done. Source files stay read-only. Re-run processes only pending/failed.
LIBS: better-sqlite3, drizzle-orm + drizzle-kit, sqlite-vec, SQLite FTS5 (built-in), ulid, DuckDB (optional, attached)
RISKS:
  - [high] EAV claim row explosion (10-30 claims/file x 1M files = tens of millions of rows) -> Narrow claim table (value as TEXT, integer field/source enums), covering index (entityType, entityId, field); prune superseded claims per retention policy; keep all hot reads on the compact ResolvedField projection (WITHOUT ROWID); batch inserts in transactions.
  - [medium] better-sqlite3 native ABI breakage across Electron/Tauri-Node upgrades -> Pin Node/Electron ABI, automate electron-rebuild in CI, ship prebuilt per platform; under Tauri run DB in a Node sidecar with a stable IPC contract decoupling the Rust shell from the native module.
  - [medium] sqlite-vec immaturity / loadable-extension packaging on three OSes -> Abstract similarity behind a repository interface; ship a pure-TS Hamming/BK-tree fallback for perceptual hashes so dedup works even if the extension fails to load; gate vector features behind a startup capability check.
  - [medium] Single-file write contention during multi-worker scans -> WAL + one dedicated writer worker; scan/fingerprint workers stream results over a channel and the writer batches them transactionally; busy_timeout + short transactions. Reads scale freely under WAL.
  - [medium] Migrations against a 1M-row live user DB can be slow/locking -> Forward-only versioned migrations with checksums; prefer additive changes; run long migrations in background with progress + checkpoint control; VACUUM/ANALYZE post-migration; schema_version table refuses to open newer DBs with older app.
  - [low] Resolver non-determinism could make resolved values flap across scans -> Resolution is a pure deterministic function of (source priority, confidence, recency, human-override) in a versioned MergePolicy; human overrides stored as a claim with source=human at max priority and always win; recompute is idempotent.
MILESTONES:
  - M1 -- Spine + scan/provenance MVP: SQLite+Drizzle setup; MediaFile + TrackFile/PhotoFile/VideoFile; MetadataClaim + ResolvedField + resolver + MergePolicy; ScanRun/ScanCursor/ScanError incremental resumable scan; exact-dup via contentHash; FTS5. Music-first. Read-only library view powered by ResolvedField.
  - M2 -- Music logical layer + reconstruction: Artist/ArtistCredit/ReleaseGroup/Release/Medium/Recording/Track; ExternalId; AcoustID/MusicBrainz enrichment writing claims; AlbumCandidate/CandidateMember + candidate generation; completeness insight.
  - M3 -- Human-in-the-loop + organize: Grouping/GroupMembership; GroupingDecision FSM (candidate->reviewed->confirmed); human-override claims; OrganizePlan/OrganizeAction copy journal with dry-run + resume.
  - M4 -- Near-dup + photo/video + insights: sqlite-vec near-dup (audio/image/video); PhotoEvent/Series/SeriesEntry + perceptual hashing; DuplicateGroup/DuplicateMember; CollectionStat/OwnerProfileSnapshot (optionally attached DuckDB).

## [reconstruction] Album / unit reconstruction engine — entity resolution over scattered audio tracks
SUMMARY: A four-stage entity-resolution pipeline rebuilds release units from scattered, mistagged, duplicated audio: (1) per-file FEATURIZATION extracts tags, Chromaprint fingerprint, folder/filename signals, and audio properties into a SQLite-backed track index; (2) BLOCKING generates candidate clusters via multiple cheap keys (folder, normalized album+albumartist, MBID, AcoustID) so we never compare all pairs; (3) ENRICHMENT/MATCHING resolves tracks to MusicBrainz recordings then disambiguates the correct RELEASE by scoring tracklist coverage, length agreement, date, country, and edition signals, with offline fallback to tag/folder clustering; (4) ASSEMBLY produces ranked album candidates with per-track confidence, best-copy dedup among bitrate variants, completeness scoring vs the chosen tracklist, and an explainable "why grouped" trace. The engine runs in a Node/TS sidecar, is fully incremental/resumable via content-hash keys, and degrades gracefully when offline.
RECO: Make the RELEASE (not the recording) the resolution target, and treat reconstruction as constrained set-cover: for each candidate cluster, pick the MusicBrainz release whose tracklist is best COVERED by the cluster's recordings (length-aware, position-aware), not the release with the most individual high-score track matches. This single inversion fixes the dominant failure mode (greedy per-track matching scatters an album across reissues/comps) and yields a natural completeness score for free. Always keep an offline path that produces the same candidate shape so the UI is identical with or without network.
DECISIONS:
  - Resolution target granularity => Resolve to a specific MusicBrainz RELEASE, with the RELEASE-GROUP as the fallback grouping when a single release can't be confidently chosen
  - Matching paradigm => Constrained set-cover: pick the release best COVERED by the cluster, then assign tracks to positions
  - Avoiding O(n^2) => Multi-key blocking + union-find with fan-out caps; pairwise scoring only inside candidate clusters
  - Online dependency posture => Hybrid with offline-first fallback producing identical candidate shape; network strictly enriches confidence
  - Identity/caching key => Content hash of decoded audio stream (not file bytes) as the stable track identity for incremental/resumable scans and dedup
  - Duplicate handling timing => Dedup AFTER release assignment, per album position, keeping alternates
ALGORITHMS:
  - Per-file featurization & stable identity: For each file: (1) parse tags+tech props via music-metadata; (2) compute a stable audioHash = hash of decoded PCM frames or, cheaper, hash(Chromaprint raw fingerprint + rounded duration) — used as track identity for incrementality and dedup; (3) run fpcalc → (fingerprint, duration) cached by audioHash; (4) tokenize folder path and filename (extract trailing/leading track numbers, disc markers like 'CD1','D2', strip junk: '[FLAC]','(2009 Remaster)'); (5) record mtime/birthtime, codec, bitrate, samplerate, lossless flag, embedded-art bool, existing MBIDs. Persist TrackFeature row keyed by audioHash; skip recompute if audioHash already present (resumable). Confidence inputs are stored, not yet scored.
  - Multi-key blocking + union-find candidate generation: Emit blocking keys per track: K1=folderId (immediate parent dir id); K2=norm(albumArtist)+'|'+norm(album) where norm = lowercase, strip diacritics, strip edition noise via regex, collapse whitespace; K3=tagged MB releaseGroup MBID; K4=AcoustID id (if resolved); K5=norm(album)+'|'+roundYear (catches missing albumartist). Build inverted index key→tracks. Merge two tracks into the same cluster via union-find ONLY when they share K3/K4 (strong, single-key merge) OR share TWO of {K1,K2,K5} (weak keys need corroboration). Enforce maxBlockSize (e.g. 400 in-block tracks); if exceeded, refine by splitting on disc/year/fingerprint-cluster. Output candidate clusters (typically one per physical album, sometimes a few editions together).
  - Track→recording resolution (online): For each track with a fingerprint, query AcoustID → list of (recordingMBID, score). Keep recordings with score ≥ 0.7. Cross-check returned recording title/length against tags: accept if titleSim(Jaro-Winkler) ≥ 0.85 OR duration within ±4s. If tags already carry a recording/track MBID (Picard-tagged), trust it as a high-prior candidate (prior weight 0.4) but still verify length. Aggregate at the cluster level the set of candidate releases referenced by the cluster's matched recordings (via MB inc=releases).
  - Release disambiguation via constrained set-cover (the core scorer): For candidate cluster C with matched recordings R and candidate releases {Rel}, score each Rel: coverage = (#positions of Rel filled by some r∈R matched by length±3s)/totalTracks(Rel); extra = #tracks in C not placeable on Rel; then ReleaseScore = w1*coverage − w2*(extraPenalty per unplaced) + w3*editionPref + w4*countryDateMatch + w5*formatMatch + w6*coverArtPresent. Weights (initial): w1=0.55, w2=0.15, w3=0.12, w4=0.08, w5=0.05, w6=0.05. editionPref = +1 official, −0.5 bootleg/promo, prefer non-compilation unless cluster artist-credit is VA, prefer earliest release-group date when 'original' intent (configurable: original vs owned-edition). Choose argmax Rel if ReleaseScore ≥ 0.6 AND coverage ≥ 0.5; else fall back to release-group grouping or offline cluster. Assign each track to its Rel position by (recording match → position) using Hungarian/greedy on (lengthDelta, titleSim, tagTrackNo agreement). Per-track confidence = sigmoid over (acoustidScore, lengthDelta, titleSim, positionAgreement, tagTrackNoAgreement).
  - Offline album reconstruction (no network): Within each candidate cluster, group by (norm albumartist, norm album, roundYear); if albumartist blank or VA, group by (folderId, norm album, year). Infer discCount from disc# tags or folder markers (CD1/CD2). Order tracks by (disc#, track#) from tags; for tracks missing track#, parse leading/trailing integer from filename; remaining unordered tracks appended and flagged. Estimate totalTracks = max(observed track#) per disc; missing = totalTracks − observed (gap detection from track-number holes); extra = duplicate track numbers or out-of-range. Completeness = observed/totalTracks (capped, confidence-discounted because totalTracks is inferred). Produce same AlbumCandidate shape with source='offline', lower base confidence (≤0.75).
  - Best-copy selection among duplicates: Within an assigned release position (or offline track slot), group copies sharing the same recording (AcoustID equal) or audioHash-similar (fingerprint similarity ≥ 0.95 AND duration ±2s). Rank by tuple: (lossless desc, bitrate desc, samplerate desc, channels desc, tagCompleteness desc, embeddedArt desc, longerDurationIfTruncated). Winner = rank-1, mark canonical; others retained as alternates with reason ('lower bitrate 320<FLAC'). This selection populates the COPY (never mutates source).
  - Confidence model & thresholds: Overall AlbumCandidate confidence = weighted blend of mean per-track confidence (0.5), release coverage (0.3), and signal corroboration count (0.2: how many independent signals — fingerprint, tags, folder, MBID — agreed). Buckets: ≥0.85 auto-acceptable (still user-confirmable), 0.6–0.85 review-recommended, <0.6 needs-review/uncertain. Per-track confidence similarly bucketed to drive UI badges. Thresholds are config-exposed; nothing is written to the destination tree below the auto-accept threshold without explicit user action (aligns with copy-only safety).
  - Explainable 'why grouped' trace: For each candidate, emit a structured evidence object: blockingKeysMatched (which keys/values co-clustered the tracks), recordingMatches (acoustid scores, lengthDeltas), chosenRelease (mbid, score breakdown per weighted term, runners-up with their scores), perTrack (assignedPosition, signals that placed it, conflicts), completeness (observed/total, missing positions, extras), dedup (collapsed copies + ranking reasons). UI renders this as a human-readable rationale and a diff against alternates. Persist trace for incremental re-scan diffing.
LIBS: music-metadata, Chromaprint / fpcalc, AcoustID web service, MusicBrainz WS/2 (JSON), Cover Art Archive, better-sqlite3, bottleneck, execa, fastest-levenshtein / talisman, undici
RISKS:
  - [high] MusicBrainz 1 req/s limit makes online resolution of 1M tracks take days if naive -> Resolve at the CLUSTER level (one release lookup per ~album, not per track); cache all responses on disk; batch AcoustID; allow an optional local MB mirror/datadump for power users; make online enrichment a background, resumable, throttled pass that never blocks offline candidate generation.
  - [high] Mega-blocks: generic folders ('Music', 'Downloads') or empty/identical album tags create huge candidate clusters → quadratic blowup inside a block -> Cap block size; require AT LEAST TWO agreeing strong keys to merge; split oversized blocks by secondary keys (date, discId, fingerprint cluster); treat blank/sentinel album tags ('Unknown Album') as non-keys.
  - [medium] Wrong release chosen (reissue/compilation/region) despite high per-track scores -> Set-cover selection + explicit edition-preference weights (prefer official > bootleg, original release-group date, matching country/format, non-compilation unless artist-credit is VA); expose the trace and a 'choose different edition' affordance; never auto-commit below the confidence threshold.
  - [medium] Classical and various-artists releases break artist/album assumptions (composer vs performer; per-track artists) -> Special-case VA: cluster on album+date+folder ignoring per-track artist; for classical, key on release-group + work/movement via MB work relations and disc structure rather than 'album artist'; flag these for review rather than forcing standard scoring.
  - [medium] Chromaprint native binary absent or wrong arch on a platform → fingerprinting silently disabled -> Bundle per-platform binaries, verify checksum + run a self-test at startup, surface a clear capability flag; degrade to offline tag/folder clustering with lowered confidence rather than failing the scan.
  - [medium] Duplicates at different bitrates picked inconsistently or true variants merged with distinct recordings -> Dedup only within an assigned release position; require fingerprint/AcoustID agreement AND duration within tolerance before collapsing; keep losers as explorable alternates with the ranking rationale.
  - [medium] Offline heuristics over-merge unrelated tracks that share a common album tag ('Greatest Hits', 'Live') -> Require folder OR albumartist agreement in addition to album tag; weight folder cohesion heavily offline; cap inferred completeness confidence and mark low-evidence clusters as 'needs review'.
  - [low] Tauri/Electron sidecar lifecycle complexity for the long-running native engine -> Run the engine as a dedicated Node sidecar process with a typed IPC contract and a job queue persisted in SQLite so crashes resume from the last committed batch; keep UI thin.
MILESTONES:
  - M0 — Index & Featurize: Sidecar process, SQLite schema, recursive scanner, music-metadata extraction, fpcalc integration + worker pool, stable audioHash, resumable/incremental scan over 100k+ files with bounded memory.
  - M1 — Offline Reconstruction: Multi-key blocking + union-find, offline cluster builder, track-order inference, completeness estimation, best-copy dedup, AlbumCandidate + EvidenceTrace output, review-UI IPC. Fully functional with zero network.
  - M2 — Online Enrichment: AcoustID resolution, MusicBrainz release fetch + persistent cache, set-cover release disambiguator, edition preferences, confidence blend, Cover Art Archive. Online becomes a throttled background upgrade pass over M1 candidates.
  - M3 — Hard Cases & Tuning: VA/compilation handling, multi-disc/box-set re-merge, classical work/movement support, live/bootleg edition flags, threshold/weight tuning harness on a labeled test corpus, alternates UI.
  - M4 — Commit & Incremental Diff: Copy-only materialization to destination tree, incremental re-scan diffing against stored candidates/traces, optional local MusicBrainz mirror support for 1M-scale users.

## [enrichment] Fingerprinting & external enrichment services (the hybrid layer)
SUMMARY: A two-tier enrichment pipeline: (1) local fingerprinting via a bundled per-OS `fpcalc` (Chromaprint) producing AcoustID-compatible fingerprints, and (2) an external-metadata resolver that fans out to AcoustID -> MusicBrainz -> Cover Art Archive (and Discogs as supplement). Every network call passes through a single content-addressed, on-disk SQLite cache with a token-bucket rate-limiter per source, exponential backoff, and request dedup, so the same logical query is never issued twice and a 1M-item run is fully resumable. At true 1M scale the design assumes a LOCAL MusicBrainz database dump/replica is the primary metadata source and the live API is a fallback/delta channel — public-API rate-limit math makes 1M live lookups impossible otherwise. All results attach to the data model as provenance-stamped, confidence-scored facts (never overwriting source files), with graceful offline fallback to embedded tags + heuristics.
RECO: Treat the public APIs as a "small-collection / delta" path and make a LOCAL MusicBrainz Postgres replica (Live Data Feed) the canonical metadata source for any library above ~50k recordings. The math is decisive: MusicBrainz allows ~1 req/s/IP, so 1M uncached recording lookups = ~11.6 days of wall-clock at the cap even at 100% throughput. AcoustID at 3 req/s is the gating fingerprint-resolution step (~3.9 days for 1M). The local dump turns metadata resolution into local Postgres joins (millions/hour), leaving the network used only for fingerprint resolution (AcoustID, unavoidable) and cover art. Architect the resolver behind a source-agnostic interface so "local replica" and "live API" are swappable backends sharing one cache and one data model.
DECISIONS:
  - How to run Chromaprint from TS => Bundle the prebuilt fpcalc CLI binary per-OS (win/mac-x64/mac-arm64/linux) and shell out via execa, rather than FFI/N-API bindings to libchromaprint.
  - Primary metadata source at scale => Local MusicBrainz Postgres replica via mbslave/Live Data Feed for libraries >~50k; live API only for small libraries and incremental deltas.
  - Caching layer technology => better-sqlite3 single-file content-addressed cache with WAL, not a KV store or filesystem-of-JSON.
  - Fingerprint dedup strategy => Cache AcoustID lookups keyed by the fingerprint hash, and pre-dedup identical audio by content hash BEFORE fingerprinting.
  - Conflict resolution / precedence => Per-field provenance with a configurable precedence (user-edit > AcoustID-linked MBID match > MusicBrainz direct > Discogs > embedded tag > heuristic), plus a numeric confidence.
  - Offline fallback behavior => Circuit-breaker per source; on sustained failure or no-network, resolver degrades to embedded-tag parsing (music-metadata) + filename/folder heuristics, marking results low-confidence and flagged for later online re-enrichment.
ALGORITHMS:
  - Rate-limit budget math at 1M scale: Inputs: N recordings, dedup ratio d (unique fraction). Live AcoustID: 3 req/s -> seconds = (N*d)/3. For N=1M, d=0.5 -> 500k/3 ~= 46.3h (~1.9 days); d=1.0 -> ~3.9 days. Live MusicBrainz: 1 req/s -> unique-release lookups U/1; if U=300k unique releases -> ~83h (~3.5 days), and naive per-recording (1M/1) ~= 11.6 days. Discogs: 60/min -> 1M/60 = ~11.6 days (so Discogs is supplement-only, never bulk). Conclusion encoded as policy: bulk metadata MUST come from the local replica (SQL, no rate limit); network budget reserved for AcoustID fingerprint resolution (irreducible) + cover art. Always batch AcoustID with meta=recordings+releasegroups to get MBIDs in one call and skip a separate MB recording fetch.
  - Content-addressed cache + in-flight dedup: key = sha256(source || endpoint || canonicalize(params)). Lookup order: in-flight promise map -> SQLite hit (incl. negative-cache TTL) -> network. On network success store {payload, status, fetched_at, etag}. Identical concurrent requests share one promise (single-flight). Negative results cached with shorter TTL so transient 404s re-resolve later.
  - Pre-fingerprint audio dedup: Compute a fast file hash (xxhash of full bytes, or of decoded-audio-stream hash for tag-insensitive matching) before fingerprinting. Group identical hashes; fingerprint+lookup the representative once, fan the resolved MBID to all members. Track duplicate clusters for the duplicates insight.
  - Release reconstruction confidence scoring: For each candidate release from AcoustID/MB, score = w1*(fraction of album's tracks matched to the same release MBID) + w2*(track-count match vs release track count) + w3*(duration agreement) + w4*(source agreement AcoustID&MB&Discogs concur) - penalties (conflicting catalog/year). Emit per-release confidence 0..1 and per-field provenance. Threshold gates auto-apply vs flag-for-review.
  - Exponential backoff with full jitter: delay = random(0, min(cap, base * 2^attempt)); base~=500ms, cap~=60s; retry on 429/503/5xx/network; respect Retry-After header verbatim if present (overrides formula). After K consecutive failures open circuit for source -> fallback path.
LIBS: fpcalc (Chromaprint), execa, musicbrainz-api (Borewit), better-sqlite3, pg (node-postgres), undici / got, bottleneck (or p-ratelimit), p-retry + cockatiel, exifr, local-reverse-geocoder, music-metadata (Borewit)
RISKS:
  - [high] Treating the live public APIs as the bulk source — at 1M items MusicBrainz (1 req/s) and Discogs (60/min) make full runs take ~1.5-2 weeks and risk IP throttling/bans. -> Hard-gate bulk metadata to the LOCAL MusicBrainz replica; cap live API to small libraries + incremental deltas. Enforce a single shared per-host limiter and persist its state across resume.
  - [medium] Bundling/notarization of the native fpcalc binary across win/mac(arm+x64)/linux, plus its LGPL/ffmpeg licensing. -> Ship official prebuilt fpcalc per target in app resources; checksum-verify at launch; include LGPL notices and source offer; codesign+notarize macOS binaries; document the FPCALC path resolution. In Tauri, run fpcalc from the Node sidecar so the Rust shell stays thin.
  - [high] Missing/invalid User-Agent or exceeding limits gets the app's IP blocked by MetaBrainz/Discogs, harming all users on that NAT. -> Mandatory descriptive User-Agent (app, version, contact) baked into clients; conservative limiter defaults; adaptive slowdown on 429/503; circuit breaker. Borewit musicbrainz-api enforces UA + throttle by default.
  - [medium] Cache corruption or non-resumable runs waste days of fingerprinting/lookups after a crash. -> SQLite WAL + transactional writes; store raw payloads separately from parsed so parsing can be re-run without re-fetch; idempotent keys; persist limiter/progress cursors for clean resume.
  - [medium] Discogs/Cover Art Archive/TMDb ToS and attribution: Discogs forbids bulk scraping & has commercial restrictions; CAA/MB are CC0 data but images vary; TMDb requires attribution. -> Use Discogs only as on-demand supplement (never bulk), respect its headers, keep token user-supplied. Prefer the downloadable MB/Discogs data dumps for bulk where licensing permits. Display required TMDb attribution; store source/license per fact for audit.
  - [low] Reverse geocoding 1M photos online would dwarf even the music rate limits. -> Use a fully local GeoNames reverse-geocoder; no network per photo.
  - [medium] AcoustID is the irreducible network bottleneck (3 req/s) even with a local MB replica. -> Aggressively pre-dedup identical audio before lookup; cache by fingerprint hash; batch lookups with full meta to avoid follow-up MB calls; run it as a long-lived resumable background job. (A local AcoustID DB is impractical; accept this as the one true online dependency.)
MILESTONES:
  - MVP (online, small library): Bundled fpcalc + FingerprintWorker; AcoustIdClient with 3/s limiter; LiveApiBackend via musicbrainz-api (MB + CAA); better-sqlite3 content-addressed cache with single-flight dedup; backoff+retry; ProvenanceMapper writing metadata_fact/external_ref; offline fallback to music-metadata tags. Works end-to-end for ~10k tracks.
  - Scale (local replica + resumability): LocalMbBackend over a provisioned MusicBrainz Postgres replica (mbslave docs/scripts); backend auto-selection; persisted limiter + progress cursors for resumable 1M runs; pre-fingerprint audio dedup; negative-cache + adaptive throttling. Demonstrate 1M-item run bounded-memory + resume-after-kill.
  - Supplement + non-music: DiscogsClient (token, header-driven 60/min, supplement-only); CoverArtClient artwork download into output tree; ImageVideoEnricher (exifr + local reverse-geocode; optional TMDb/OMDb). Confidence scoring + flag-for-review UI hooks; per-source attribution/license surfaced.

## [organize] Metadata editing + "organize into a NEW TREE (copy)" engine
SUMMARY: A read-only-source, copy-to-new-tree engine built around three pillars: (1) a unified TagWriter abstraction over node-taglib-sharp (pure-TS, no native deps, writes ID3v2.3/2.4, Vorbis comments, MP4 atoms, and embeds cover art uniformly), with mp3tag.js as a narrow fallback for ID3 edge cases; (2) a pure, deterministic path-templating engine with cross-platform sanitization, NFC unicode normalization, and per-component byte-length clamping; (3) a transactional, journaled OrganizePlan executor: dry-run preview -> confirm -> execute, where every file is hash-verified after copy and every action is appended to an idempotent, resumable journal. The destination is always reproducible from source + plan + journal; no source mutation and no source-undo is ever needed. Tag writes happen on the COPY (copy first, then tag, then verify), never on the original.
RECO: Tag the COPY, never the source: copy raw bytes -> verify copy hash matches source -> open the copy and write tags/art in place -> record both source hash and final-copy hash in the journal. Drive the whole thing from a serialized OrganizePlan (JSON) whose execution is journaled per-action so it is fully resumable and idempotent. Standardize on node-taglib-sharp as the single cross-format writer because it is pure TypeScript with zero native dependencies — this is the deciding factor for desktop packaging (no node-gyp, no prebuild matrix, works identically under an Electron Node engine or a Tauri Node/TS sidecar).
DECISIONS:
  - Primary cross-format tag-writing library => node-taglib-sharp (pure-TS TagLib# port) as the single writer for ID3v2.3/2.4, Vorbis comments (FLAC/Ogg/Opus), and MP4 atoms (M4A/ALAC/AAC), including cover-art embedding
  - How tags are written without touching source => Copy bytes first, verify copy hash equals source hash, THEN open the COPY and write tags + art in place on the copy
  - Hash algorithm for verification => blake3 for integrity verification, xxhash (xxhash-wasm) for fast dedup pre-screening
  - Journal storage => better-sqlite3 in WAL mode as the source of truth, with an NDJSON mirror for portability/audit export
  - Verify-after-copy granularity => Hash the written copy and compare to the pre-recorded source hash on EVERY file; store both hashes in journal
  - Concurrency model for copy + tag at scale => Bounded worker pool (piscina) with separate, independently-tunable concurrency for I/O-bound copy/verify vs CPU-bound tag/art encode; default copy concurrency low (2-4) on spinning disks, higher on SSD/NVMe
ALGORITHMS:
  - Path templating + cross-platform sanitization: 1) Resolve tokens (AlbumArtist/Album/Year/Disc/Track/Title) from the canonical AlbumUnit model with fallbacks (AlbumArtist -> Artist -> 'Unknown Artist'; missing Year -> omit token segment). 2) NFC-normalize all unicode. 3) Per path COMPONENT: replace illegal chars (<>:"/\|?* and control chars) with a configurable substitute, strip trailing dots/spaces, reject Windows reserved device names by appending '_'. 4) Clamp each component to a byte budget (target ~255 bytes per component, NFC-aware so multibyte chars aren't split) and clamp total path to a conservative cross-platform limit (default ~255 chars unless long-path mode is enabled on Windows). 5) Zero-pad track (NN) and disc to template width. Engine is a pure function for full test coverage.
  - Multi-disc folder layout: If AlbumUnit.discCount > 1: choose layout per config — either flat with disc-prefixed track tokens (e.g. '1-01 - Title') OR nested 'Disc N/NN - Title'. Track numbering derives from per-disc track index; total-disc/total-track tokens populated for tag frames (TPOS/discnumber/disk atom).
  - Copy + verify pipeline (per file): 1) Preflight: ensure free space >= sourceSize * safetyFactor, dest path length legal, parent dirs created. 2) Compute/lookup source hash (reuse scan-phase hash if available to avoid re-read). 3) Copy bytes (fs.copyFile for same-volume speed, else streamed copy with backpressure). 4) Re-hash the COPY; compare to source hash -> on mismatch, delete copy, mark action failed, retry up to N. 5) Open copy with TagWriter, apply tagDelta + embed art, save. 6) Optionally re-hash/re-read to record final tagged-copy hash + re-read tags for assertion. 7) Journal action=done with sourceHash, copyHash, finalHash, tagVersion, bytes, durations.
  - Idempotent resume: On execute, load journal; for each plan action compute desired state. Skip actions already 'done' whose recorded sourceHash still matches current source (re-stat + cheap xxhash if size/mtime changed). For 'copied-not-tagged' resume at tag step; for 'failed' retry. Because dest is derived purely from (plan, source), re-running converges to the same tree.
  - Collision + duplicate resolution: Build an in-memory map of destPath -> action during planning. On collision: if both source hashes equal -> mark as duplicate (skip copy, optionally record alias); else apply policy (numeric suffix ' (2)', or route to a 'Duplicates/' quarantine subtree). Content hash, not name, is the duplicate authority.
  - Free-space + throughput governance: Before execute: sum plan bytes, compare to target free space + headroom. During execute: periodically re-check free space; throttle/pause if below threshold. Track rolling MB/s to estimate ETA and to auto-tune copy concurrency (raise on SSD-like throughput, cap on HDD-like).
LIBS: node-taglib-sharp, music-metadata, mp3tag.js, better-sqlite3, piscina, xxhash-wasm, blake3 (blake3-wasm or hash-wasm), check-disk-space, filenamify (or sanitize-filename), graceful-fs, zod
RISKS:
  - [medium] node-taglib-sharp write fidelity on exotic/edge files (odd ID3 frames, non-standard MP4 atoms, broken Vorbis blocks) could produce tags some players misread -> Always read-back-and-assert after write (re-parse the copy with music-metadata and compare to intended model); maintain a format conformance test corpus; keep mp3tag.js / metaflac CLI as a per-format escape hatch for the small failing set; quarantine and journal files that fail assertion rather than silently shipping bad tags.
  - [medium] better-sqlite3 is a native module — Electron ABI mismatches or Tauri sidecar packaging can break the journal store -> Pin Electron + better-sqlite3 versions and run electron-rebuild in CI; for Tauri, isolate it in the Node sidecar with a known Node version; keep the NDJSON mirror so the journal survives even if SQLite fails to load, enabling degraded-mode resume.
  - [high] Cross-platform filename correctness bugs (Windows reserved names, case-insensitive collisions, path-length limits) silently corrupt the output tree -> Pure-function PathTemplateEngine with an exhaustive cross-platform unit-test matrix (reserved names, trailing dot/space, 255-byte vs 255-char, surrogate pairs, case folding); validate ALL paths during dry-run before any write; opt-in Windows long-path (\\?\) mode.
  - [high] Performance/handle exhaustion at 1M files (EMFILE, memory blowup, HDD thrash) -> graceful-fs + bounded piscina pools with separate I/O vs CPU concurrency; stream don't buffer; never Promise.all the world; reuse scan-phase hashes to avoid double full-reads; batch journal writes in transactions.
  - [medium] Partial failure / crash mid-run leaves a half-built tree -> Append-only journal with per-step states + atomic temp-file-then-rename for copies (write to .part, fsync, rename) so a crash never leaves a file that looks complete; idempotent resume converges to the intended tree; orphan .part cleanup on restart.
  - [medium] Disk fills mid-run on TB-scale copy -> Preflight total-bytes vs free-space with headroom; periodic free-space checks during run; pause-and-surface (not crash) when below threshold; per-album transactional grouping so a pause leaves whole albums, not half-albums.
  - [low] Cover-art bloat and inconsistency (huge embedded JPEGs across thousands of tracks) -> Normalize art on embed: re-encode to a configurable max dimension/quality (sharp), dedupe identical art by hash, embed once-per-track consistently; optionally also drop a folder.jpg at album root.
MILESTONES:
  - M1 - MVP single-format copy+tag: PathTemplateEngine (pure, tested) + CopyPipeline with blake3 verify + node-taglib-sharp writing ID3v2.4 for MP3 only; basic JSON OrganizePlan; sequential execution; read-back assertion. Proves the copy-then-tag-the-copy safety model end to end.
  - M2 - All formats + cover art: Vorbis (FLAC/Ogg) and MP4 (M4A/ALAC) writers via the same facade; cover-art embedding with sharp normalization + folder.jpg; CollisionResolver with content-hash dedup; multi-disc layouts.
  - M3 - Journal, dry-run, resume: better-sqlite3 journal (WAL) + NDJSON mirror; full dry-run preview with collision/disk/path validation; idempotent resume; auditable journal export.
  - M4 - Scale + resilience: piscina worker pools with split I/O/CPU concurrency; graceful-fs; free-space governance + ETA/throughput auto-tuning; atomic .part-then-rename; partial-failure recovery + retries; large-corpus soak test (>=1M synthetic files).
  - M5 - Polish: Metadata-edit UI bindings (batch edits -> plan), format conformance corpus + player-compat assertions, long-path Windows mode, configurable template token library.

## [collection-insights] Collection analytics — "what the collection itself says"
SUMMARY: An analytics engine that treats the catalog as a columnar OLAP problem layered on the same embedded store used by the scanner. At 1M+ items, live GROUP BY scans are too slow for interactive dashboards, so the design splits into two tiers: (1) cheap, always-live aggregates served by SQL over indexed columns (counts, byte sums, format breakdown), and (2) precomputed, incrementally-maintained rollup tables (genre distribution, decade histograms, completeness, duplicate clusters, owner-profile timelines) refreshed via a dirty-set delta after each scan batch rather than full recomputation. Heavy DSP work (spectral cutoff / fake-FLAC detection) runs only on a sampled subset in the Node/TS engine, with results cached per content-hash so they survive re-scans. All metrics are versioned by a schema/rollup epoch so cached rollups invalidate cleanly when computation logic changes.
RECO: Use DuckDB (via duckdb-async / @duckdb/node-api) as the analytics engine sitting alongside the SQLite operational catalog — DuckDB's columnar/vectorized execution makes full-collection aggregations (genre, decade, bitrate histograms) interactive at 1M rows on a desktop, while SQLite remains the transactional store for scan state and edits. Maintain materialized rollup tables refreshed incrementally from a dirty-row set, never full recompute. Treat spectral/transcode detection as an opt-in, sampled, content-hash-cached pass — never block dashboard rendering on DSP.
DECISIONS:
  - Analytics storage engine => DuckDB for OLAP aggregation, alongside SQLite for the operational/transactional catalog
  - Rollup strategy => Incrementally-maintained materialized rollup tables driven by a dirty-row set, full rebuild only on rollup-epoch bump
  - Transcode / fake-FLAC detection scope => Opt-in, sampled, content-hash-cached spectral analysis — not run on every file by default
  - Genre source-of-truth ordering => Curated MusicBrainz genre > MB folksonomy tag (vote-weighted) > Discogs style > local tag string
  - Owner-profile time signal => Use filesystem birthtime where available, fall back to mtime, captured once at scan time and stored as a column
ALGORITHMS:
  - Style/genre distribution (normalized): 1) For each track collect candidate genres from sources with weights: MB curated genre w=1.0, MB folksonomy tag w=tag_votes-normalized to 0.4-0.8, Discogs style w=0.6, local tag string w=0.3. 2) Normalize each raw string: lowercase, strip punctuation, split on [/,;|&] and 'feat'-style noise into multiple genres, map through alias table (e.g. 'hiphop'->'hip hop', 'drum n bass'->'drum and bass', 'alt rock'->'alternative rock'); unmapped strings get fuzzy-matched (Jaro-Winkler) to canonical set with threshold >=0.92 else bucketed as 'unmapped'. 3) Resolve per track to top-N weighted genres (default top 2) to avoid double counting; emit (track_id, genre, weight). 4) Rollup table genre_counts(genre, track_count, weighted_count, byte_sum) maintained incrementally. Distribution = weighted_count share; also expose 'primary genre' mode for clean pie. Confidence per genre = sum of source weights / max.
  - Total volume (count / bytes / hours): Live SQL: SELECT count(*), sum(size_bytes), sum(duration_sec)/3600 FROM fact_track [WHERE media_type='audio']. Hours of audio uses duration captured at scan (ffprobe/music-metadata). Maintain a tiny single-row rollup for instant header KPIs updated by delta on each batch so the very first dashboard paint needs zero scan.
  - Format & lossless breakdown: Classify each file by codec into {lossless: FLAC, ALAC, WAV, AIFF, APE, WavPack, TAK; lossy: MP3, AAC, OGG/Vorbis, Opus, WMA, MQA-flagged}. Lossless flag derived from codec, not extension (catch .m4a = ALAC vs AAC by probing codec_name). Rollup format_counts(codec, lossless_bool, track_count, byte_sum, avg_bitrate). Bit depth & sample rate from stream metadata -> 2D histogram (e.g. 16/44.1, 24/96). Bitrate histogram bucketed (lossy: <128,128,160,192,256,320,VBR-bins; lossless: derived kbps). Lossless% = lossless byte_sum and track_count shares (report both — count vs size tell different stories).
  - Quality / transcode (fake-FLAC, upscaled lossy) detection: For sampled files (all lossless by default, optional all): decode N representative windows (e.g. 6x 4s segments across the track) to mono PCM via ffmpeg; for each window compute power spectrum (FFT, Hann window), average into a long-term spectrum; find the highest frequency bin whose energy exceeds noise floor (cutoff_hz) via a sliding-threshold from the top (energy drops >40 dB and stays). Decision: lossless file with cutoff clustering near 16kHz=>transcoded from ~128-160 MP3, ~19kHz=>~192-256, ~20-20.5kHz with sharp brickwall=>likely lossy-sourced; genuine lossless shows energy to ~22 kHz (44.1) or higher with no brickwall. Score = confidence(0-1) from cutoff sharpness + plateau. Lossy upscale: file claims 320 but real cutoff <16kHz => flag. Cache result by content_hash; store cutoff_hz, verdict, confidence in track_quality table.
  - Duplicate analysis: Three-tier union-find clustering. Tier A exact: GROUP BY content_hash (already computed by scanner). Tier B acoustic: pairs sharing AcoustID/Chromaprint fingerprint (or fp similarity above threshold) — same recording, different encode. Tier C metadata-loose: DuckDB self-join on normalized(artist)+normalized(title) with abs(duration_a-duration_b)<=2s. Merge tiers with union-find into clusters tagged by strongest tier. Per cluster compute wasted_bytes = sum(size) - max-quality-copy size, and rank 'keep' copy by (lossless>lossy, higher bitrate, has cover art, better tag completeness, in a complete album). Rollup: dup_summary(total_clusters, redundant_files, wasted_bytes by tier).
  - Album completeness: Per reconstructed album: expected = MB release totaltracks (and per-disc counts) if matched, else inferred = max(tracknumber) per disc. present = distinct tracknumbers found. completeness_pct = present/expected. Gaps = expected set minus present set (list missing NN). Classify complete (100%), near-complete (>=90%), partial, fragment(<=2 tracks), singleton/orphan. DuckDB window over (album_id, disc_no) generate_series to find missing numbers. Rollup album_completeness(album_id, expected, present, pct, status) + collection histogram of statuses.
  - Top artists / decades: Top artists: rollup artist_stats(artist_id, track_count, album_count, byte_sum, total_hours); use MB artist id when resolved else normalized albumartist to merge 'The Beatles'/'Beatles'. Decades: derive year from MB original release date (preferred) else tag year; bucket floor(year/10)*10; decade histogram by track_count and by byte_sum. Cross-tab decade x genre for taste maps.
  - Cover-art coverage: Per album/track flag has_art = embedded picture frame present OR folder cover image (cover.jpg/folder.jpg) present OR Cover Art Archive available for matched MB release. Coverage% at album granularity (more meaningful than track). Also grade art quality: resolution buckets (<300px low, 300-600 ok, >=1000 high) and embedded-vs-external. Rollup art_coverage(album_id, has_art, source, max_dim).
  - Tagging hygiene score: Per track compute a weighted completeness/consistency score over required fields: title, artist, albumartist, album, tracknumber, totaltracks, date, genre, MBIDs, cover art, plus consistency checks (albumartist consistent across album, no track# collisions, no 'Unknown'/'Track 01' placeholders, charset sane). Per-track score 0-100; album score = mean with penalties for inconsistency; collection hygiene = distribution + median. Maintain rollup and a prioritized 'worst offenders' list. Weights configurable in MetricRegistry.
  - Owner profiling from build history: acquisition timeline: bucket files by stored fs birthtime (fallback mtime) via date_trunc('month') -> added-per-month series (heatmap/calendar). cadence: detect bursts (rip sessions) vs steady accrual via rolling stddev of monthly counts. classic-vs-new: per track delta = acquisition_year - release_year; classify 'collector of classics' if median delta large, 'release-chaser' if delta near 0; report classic% (release_year < acquisition_year-15). taste evolution: stacked genre share over acquisition-time buckets. Quality trajectory: lossless% over acquisition time (did they get audiophile later?).
LIBS: duckdb-async / @duckdb/node-api, better-sqlite3, music-metadata, ffmpeg/ffprobe (fluent-ffmpeg + ffprobe-static / @ffmpeg-installer), fft.js (or dsp.js / kissfft-wasm), Chromaprint / fpcalc (already in the enrichment facet), fastest-levenshtein / string-similarity (Jaro-Winkler via talisman), echarts-for-react (or visx), @tanstack/react-table + @tanstack/react-virtual, zod
RISKS:
  - [high] Full-collection aggregations stall the UI at 1M rows if served by live SQLite queries instead of DuckDB rollups. -> DuckDB columnar engine + precomputed incremental rollups; header KPIs from a single-row summary; long-tail queries streamed/paginated. Never block first paint on a full scan.
  - [high] Spectral / fake-FLAC detection is CPU-prohibitive and produces false positives on legitimately band-limited masters. -> Sampled + opt-in + content-hash cached; require sharp brickwall+plateau (not mere low cutoff) and down-weight confidence for old recordings; always present as 'suspected', never destructive.
  - [high] Build-history timestamps destroyed by the COPY-to-new-tree organize step, breaking all owner-profiling metrics. -> Snapshot source filesystem birthtime/mtime during scan into a dedicated column before/independent of organize; profile off the snapshot, not live fs.
  - [medium] Native-addon (DuckDB, better-sqlite3, ffmpeg) ABI/packaging breakage across OSes and Electron/Tauri runtimes. -> Run the analytics engine in the Node/TS sidecar (clean fit for Tauri v2 sidecar; or Electron utility process), pin prebuilt binaries, electron-rebuild in CI, and matrix-test Win/macOS/Linux packaging.
  - [medium] Genre normalization is subjective; a wrong/over-aggressive alias table yields misleading distributions the user won't trust. -> Deterministic source precedence, visible 'unmapped' bucket, user-editable alias table, and show confidence so low-confidence genres are visually distinct.
  - [medium] Duplicate clustering false-merges remasters/live versions, risking data loss if acted on. -> Confidence-banded clusters, duration tolerance, fingerprint+metadata corroboration, and copy-only/no-auto-delete policy — analytics recommends, the safe-organize facet executes with confirmation.
  - [medium] Rollup drift: incremental deltas diverge from truth over many partial scans/edits. -> Rollup-epoch versioning, periodic background full-rebuild reconciliation, and a checksum/row-count audit comparing rollup totals to base-table totals on app start.
MILESTONES:
  - M0 MVP — live KPIs + cheap breakdowns: DuckDB attached to catalog; summary_kpis rollup (count/bytes/hours), format & lossless breakdown, bitrate + sample-rate/bit-depth histograms, top artists/decades. All from indexed columns already captured at scan. Header dashboard renders instantly.

Incremental RollupMaintainer with dirty-set deltas.

## [owner-profiling] Owner profiling — "what the collection says about its owner"
SUMMARY: A fully-local inference engine that derives a psychological/behavioral portrait of the collection owner from two independent axes: (1) BUILD HISTORY — when files entered the collection, reconstructed from filesystem timestamps clustered into import batches and collecting eras; and (2) TASTE/CONTENT — what the music is (genre, era, popularity, audio quality), aggregated from the reconstruction+enrichment layers. The engine cross-correlates acquisition-time against release-year to compute classic-vs-new orientation, derives breadth/depth and completionist signals from artist/album coverage, and synthesizes a "collector archetype." Every output carries an explicit confidence score and provenance, because much of this is sensitive probabilistic inference from noisy timestamps. It runs as a pure, deterministic post-pass over the already-built SQLite index — no network, no file mutation, no telemetry.
RECO: Treat profiling as a deterministic, versioned, pure-function pipeline over the indexed catalog (SQLite → typed aggregates → derived metrics → archetype), NOT as an ML black box. Every metric must be (a) explainable to the user in one sentence, (b) tagged with a numeric confidence derived from data completeness and signal reliability, and (c) reproducible from a frozen snapshot. The single highest-leverage design choice is to make BUILD-HISTORY robust to the filesystem timestamp lie: never trust a single timestamp field; reconstruct acquisition time from a fused estimator (min of birthtime/mtime per file, denoised, then density-clustered into import batches), and degrade gracefully to "unknown era" rather than fabricating a timeline. Confidence and privacy are first-class outputs, not footnotes.
DECISIONS:
  - Which timestamp represents 'when the owner acquired this file'? => Fused estimator = min(birthtime, mtime), with ctime used only as a sanity bound; never trust a single field. Flag low-reliability cases (mtime==birthtime exactly, values at epoch 0 or far-future, 2s-granularity FAT artifacts, timestamps preceding the file's own release year).
  - How to define import batches / collecting eras? => 1-D density clustering (DBSCAN-style) over fused acquisition times: a 'burst' = many files within a tight window (e.g. >N files within <X hours); an 'era' = coarser multi-month clusters separated by dormancy gaps. Parameters are adaptive to total volume.
  - Classic-vs-new metric definition. => Per track lag = acquisitionYear - releaseYear. classic% = share where releaseYear <= (acquisitionYear - T) (default T=10y, user-tunable); new% = share where lag <= 1y. Report the full lag distribution + median, not just the binary split.
  - Breadth vs depth measurement. => Breadth = Shannon entropy over normalized genre distribution + count of distinct genres/decades. Depth = concentration via Gini/HHI of tracks-per-artist (and albums-per-artist).
  - Mainstream vs obscure signal. => Proxy from MusicBrainz/Discogs popularity signals captured during enrichment: MB artist/release-group rating count, presence in many releases, AcoustID match density. Aggregate to a percentile-based 'obscurity index'. Degrade to 'unknown' offline.
  - Archetype model: ML vs rules. => Transparent weighted-score rule model over normalized sub-metrics, emitting ranked archetypes with contributing evidence.
  - Where does the engine run and on what data? => Pure pass over a frozen SQLite snapshot in the TS engine process, after scan+reconstruct+enrich. No direct filesystem walking except the stat-fusion step (which can piggyback on the scanner's existing stat calls).
ALGORITHMS:
  - AcquisitionTime fusion (per file): Input: birthtimeNs, mtimeNs, ctimeNs (bigint), plus the file's resolved releaseYear (from enrichment) if available. Steps: 1) candidates = [birthtime, mtime] where >0 and not absurd (not epoch 0, not > now+1d). 2) acqTime = min(candidates); if none valid, acqTime = mtime (last resort) with source='weak'. 3) reliability flags: 'copied' if birthtime>mtime; 'fat-granularity' if mtime%2000ms==0 and birthtime unset; 'pre-release' if acqYear < releaseYear (impossible → clamp acqTime to releaseYear, mark 'clamped'). 4) acqConfidence in [0,1]: 1.0 when birthtime present & < mtime & plausible; 0.6 mtime-only; 0.3 weak/clamped. Output {acqTimeMs, acqConfidence, acqSource}.
  - Import-batch & era clustering: Input: sorted acqTimeMs array (streamed from SQLite ORDER BY). 1) Compute gaps between consecutive times. 2) BURST detection: a maximal run where consecutive gaps < gapBurst (adaptive: ~p10 of gaps, floored at a few minutes) AND run size >= minBurst (scaled by collection size). 3) ERA detection: merge bursts/singletons separated by gaps < gapEra (e.g. > 30–60 days = era boundary). 4) For each era/batch emit {start,end,count,medianAcqConf,density}. 5) Classify global cadence: bursty if top-K batches cover >X% of files; steady if Gini of monthly counts is low. Memory-bounded: single linear pass, no full sort in JS if SQL returns ordered.
  - Classic-vs-new lag analysis: For each track with both acqYear and releaseYear: lag = acqYear - releaseYear. Build lag histogram. classicPct = |{lag >= T}| / N (T default 10y). newPct = |{lag <= 1}| / N. medianLag, p90Lag. nostalgiaScore = sigmoid(medianLag normalized); chasingNewScore = newPct. Weight each track by min(acqConfidence, releaseYearConfidence); report effective N (sum of weights) so user sees how much data backs the claim.
  - Breadth vs depth scoring: Breadth: normalize genre counts to probabilities p_i; breadthEntropy = -Σ p_i log p_i, normalized by log(numGenres) → [0,1]; also distinctGenres, distinctDecades. Depth: tracksPerArtist distribution → Gini coefficient and HHI = Σ share_i^2; high HHI/Gini = deep/concentrated. Combine into quadrant: (wide-shallow sampler, wide-deep omnivore, narrow-deep specialist, narrow-shallow casual).
  - Completionist vs sampler index: For each primary artist with MB linkage: ownedReleases / knownStudioReleases (filter to studio albums via MB release-group type to avoid penalizing for not owning every live/compilation). albumCompleteness = ownedTracks / officialTrackCount per owned release. completionistIndex = weighted mean of (per-artist coverage) weighted by tracks owned, blended with mean albumCompleteness. Sampler = many artists with 1–2 tracks each (long tail in tracksPerArtist).
  - Audiophile signal: losslessRatio = lossless tracks / total (FLAC/ALAC/WAV/APE vs MP3/AAC/OGG). hiResShare = tracks with sampleRate>48000 OR bitDepth>16, among lossless. lossyQuality = distribution of bitrate buckets (e.g. <192/192-256/256-320/320+). audiophileScore = w1*losslessRatio + w2*hiResShare + w3*(share of lossy that is >=256k), weights normalized. Report each component separately, not just the composite.
  - Obscurity / mainstream index: Per artist/release, popularity proxy = normalized blend of MB rating-count, MB tag/relationship richness, and number of releases catalogued. Map each owned item to a percentile within the global MB-popularity reference (precomputed or relative within the collection). obscurityIndex = 1 - mean(popularityPercentile). Offline → emit 'unknown' with confidence=0.
  - Archetype synthesis: Normalize sub-scores to [0,1]: {classicPct,newPct,nostalgiaScore,breadthEntropy,depthGini,completionistIndex,audiophileScore,obscurityIndex,burstiness}. Define archetypes as weight vectors over these axes, e.g. 'The Curator/Completionist' (high completionist, high depth, high audiophile), 'The Magpie/Sampler' (high breadth, low depth, high singles tail), 'The Time Capsule/Nostalgist' (high classic%, high nostalgiaScore), 'The Early Adopter' (high newPct, low lag, bursty around release windows), 'The Archivist/Migrator' (dominant single import batch, high lossless), 'The Crate-Digger' (high obscurityIndex, high breadth). score_a = cosine(profileVector, weight_a). Return ranked archetypes with top contributing axes as human-readable evidence. Overall profile confidence = min/weighted-mean of contributing metric confidences.
LIBS: node:fs/promises (stat with bigint:true), better-sqlite3, simple-statistics, date-fns (or Temporal when stable), d3-array (optional, scoped import), zod
RISKS:
  - [high] Filesystem timestamps are systematically unreliable (copies, cloud sync, ZIP extraction rewrite mtime/birthtime), so the entire build-history timeline can be fiction. -> Fused min() estimator + reliability flags + release-year lower-bound clamp; detect single-import-day collapse and label as migration; attach per-era confidence; refuse to render a timeline when reliability is broadly low and tell the user why.
  - [high] Sensitive personal inference (taste, nostalgia, archetype) feels invasive or could be exfiltrated. -> 100% local, no network in the profiling pass, no telemetry; explicit opt-in before any profile export; one-click 'delete profile'; show confidence and 'why' for every claim; never phrase outputs as definitive judgments.
  - [medium] Missing/garbage metadata (untagged, mistagged, no MB match) biases genre/era/popularity metrics. -> Confidence weighting by data completeness; exclude-rather-than-guess for unmatched items; surface 'effective N' and coverage% next to each metric; suppress metrics below threshold.
  - [medium] Performance/memory at 1M items if profiling pulls full rows into JS. -> Do all distribution math via SQL GROUP BY/aggregates; stream ordered timestamps for clustering in a single linear pass; cache aggregates keyed by snapshot_hash so re-running with new thresholds is cheap.
  - [medium] Completionist metric unfairly penalizes owners for not having live albums/compilations/bootlegs. -> Scope 'known catalog' to studio release-groups via MB type filters; report coverage with the denominator visible; treat unmatched artists as unknown not incomplete.
  - [low] Archetype labels could feel reductive, judgmental, or just wrong. -> Return ranked multi-archetype shares with evidence, not a single verdict; user-tunable thresholds (e.g. classic cutoff T); fall back to 'insufficient data' under confidence floor.
MILESTONES:
  - MVP (offline-only, tag-based): TimestampFusion + acquisition table; BuildHistoryReconstructor (timeline, bursts vs steady, eras); ClassicVsNew using tag_year; AudiophileAnalyzer (lossless/hi-res/bitrate); breadth/depth from existing tags; per-metric confidence; persisted versioned owner_profile. No network needed.
  - Enriched profiling: Wire in MusicBrainz/Discogs enrichment columns: release-year correction via release-group first-release-date, completionist coverage vs studio catalog, album completeness, obscurity/mainstream index. Replace tag_year with MB-corrected release_year where available.
  - Archetype synthesis + UX: ArchetypeSynthesizer with ranked archetypes + evidence; confidence/privacy guard wiring; explainable per-metric 'why' strings; export-with-consent and forget-profile.
  - Hardening at scale: SQL-view aggregation, snapshot-hash caching, incremental re-profiling on rescan, migration-day/timestamp-collapse heuristics, fixture-based unit tests for every algorithm and edge case.

## [visual-media] Images & video specifics (non-music media): reconstruction analog, metadata extraction, perceptual dedup, event/series grouping, copy-organize trees, and visual-media insights.
SUMMARY: The album-integrity analog for visual media is two-tier: for PHOTOS it is the EVENT (a time+geo+burst cluster across scattered folders), and for VIDEO it is the SERIES/episode or the home-video EVENT. The pipeline mirrors music: a fast metadata-extraction pass (exiftool-vendored for EXIF/XMP/IPTC, ffprobe via static binary for video containers/codecs/HDR), then a perceptual-fingerprint pass (sharp/libvips → dHash + pHash for images; keyframe sampling → frame hashes for video) feeding dedup and grouping, then deterministic copy-organize into a date-based tree (photos) or series-based tree (video). All heavy work runs in a Node/TS worker pool (or Tauri Node sidecar), is incremental/resumable via a content-addressed catalog, and degrades gracefully offline (everything here except reverse-geocode and optional metadata lookup is fully local). Native deps are the chief packaging concern: sharp (libvips), exiftool (Perl/binary), and ffmpeg/ffprobe binaries must be unpacked outside asar (Electron) or shipped as Tauri sidecars.
RECO: Treat the PHOTO EVENT (adaptive time-gap + optional geo cluster) as the first-class "album" analog and build the whole visual pipeline as a deterministic two-pass system: Pass 1 = cheap metadata + fast dHash (sort/group/exact-dup), Pass 2 = expensive pHash + video keyframe hashing only on candidates surfaced by Pass 1. Do NOT hash every frame of every video or compute pHash on 1M images up front — gate expensive work behind cheap signals (size buckets, dHash blocks, ffprobe duration/resolution) to keep the 1M-item scale tractable and resumable.
DECISIONS:
  - Image decode + perceptual hashing engine => sharp (libvips) for all decode/resize/hash; dHash as first pass, pHash (DCT) as confirm pass
  - EXIF/XMP/IPTC extraction library => exiftool-vendored with persistent -stay_open process
  - Video probing & frame extraction => Call ffprobe-static / ffmpeg-static binaries directly with JSON output; avoid fluent-ffmpeg for the hot path
  - Photo 'album' analog => EVENT = adaptive time-gap clustering (gap grows with local density) + optional GPS DBSCAN, with burst & RAW+JPEG & live-photo sub-grouping
  - Near-duplicate search structure at scale => BK-tree (or banded LSH) over Hamming-space hashes rather than O(n^2) pairwise
  - Video content-dedup signature => Multi-keyframe perceptual hash (sample frames at fixed % offsets) + duration bucket, require majority-frame agreement
  - Catalog / resumability store => SQLite via better-sqlite3, content-addressed by (path, size, mtime, partial-hash); shared with music facet
ALGORITHMS:
  - Adaptive time-gap event clustering (photo 'album' analog): Sort items by best-confidence timestamp. Walk in order; start a new EVENT when the gap to the previous item exceeds an adaptive threshold = clamp(median_gap_of_recent_window * k, MIN=20min, MAX=12h). When GPS present, also split if great-circle distance jump exceeds a speed-implausible bound for the elapsed time. Optionally refine with DBSCAN over (lat,lng) within a day. Label event by reverse-geocoded dominant location + date span. Confidence = fraction of members with high-confidence timestamps.
  - Two-tier near-duplicate detection (dHash block → pHash confirm): Pass 1: compute exact content hash (xxhash of bytes) for true dups; compute 64-bit dHash, insert into BK-tree keyed by Hamming. Candidate near-dups = BK-tree radius query (Hamming<=10). Pass 2: for each candidate pair compute 64-bit pHash; confirm near-dup if pHash Hamming<=threshold (default 8) AND aspect ratio within 2%. Cluster via union-find. Keeper = max(resolution, then bit depth, then file size, then RAW>lossless>lossy).
  - Video content fingerprint & dedup: From ffprobe duration, sample keyframes at offsets {2%,25%,50%,75%,98%}; extract each via ffmpeg -ss seek, pipe to sharp, compute pHash → ordered vector of 5 hashes + duration bucket. Two videos are content-dups if duration within ±2% AND >=3 of 5 frame-hash pairs have Hamming<=8 in order. Keeper = max(resolution, then bitrate, then codec efficiency rank, then size).
  - Home-video vs ripped-media classification: Score with weighted signals: presence of camera GPS/creation_time atoms + device make/model (+home), filename scene-release tokens / SxxExx / 'BluRay'/'WEBRip' (+ripped), duration ~episode/movie length + standard broadcast resolution (+ripped), variable fps / phone codecs / portrait orientation (+home), multiple audio/subtitle tracks (+ripped). Threshold the score; expose as a confidence label.
  - Screenshot / scan / non-photo classification: Flag as screenshot if: no EXIF camera make/model AND dimensions match a known device/display resolution AND (filename matches /screenshot|screen shot|capture/i OR software tag indicates OS screenshot). Flag as scan if scanner software tag or no GPS+flat color profile+document aspect. Route these OUT of photo events into a separate destination subtree.
  - Timestamp resolution & provenance chain: For each item pick timestamp by priority: EXIF DateTimeOriginal/CreateDate > XMP DateCreated > video creation_time atom > filename-embedded date (regex for YYYYMMDD/unix) > filesystem birthtime > mtime. Store chosen value + source + confidence. Owner build-history insights use ONLY EXIF/embedded sources (capture time), never copy-polluted mtime.
LIBS: sharp, exiftool-vendored, ffprobe-static / ffmpeg-static, blockhash-core, sharp-phash, parse-torrent-name / anitomy-style tokenizer, better-sqlite3, node:worker_threads (built-in), local-reverse-geocoder (offline) + optional Nominatim/Photon online
RISKS:
  - [high] Native dependency bundling (sharp/libvips, exiftool, ffmpeg/ffprobe) breaks at package time — binaries inside asar can't execute, wrong arch ships, or HEIC/RAW codecs missing in prebuilt. -> asarUnpack all three (Electron) or declare as Tauri externalBin sidecars; resolve real paths at runtime; CI matrix build+smoke-test (decode a HEIC, probe an HDR mp4, read a RAW EXIF) on win/mac/linux x64+arm64 before release.
  - [high] Perceptual dedup false positives delete/mis-merge distinct images (e.g. similar landscapes, black video intros) — data-loss-adjacent if user trusts auto-merge. -> Copy-only architecture means originals are never destroyed; dedup is a non-destructive REPORT with a chosen keeper; require user confirmation; tune Hamming thresholds conservatively and require multi-frame agreement for video; always keep highest-quality variant.
  - [medium] Wrong/missing timestamps wreck event clustering and owner build-history insights (EXIF missing, timezone-naive, filesystem mtime polluted by past copies). -> Timestamp priority chain: EXIF DateTimeOriginal > XMP > video creation_time > filename date pattern > file mtime (flagged low-confidence); record provenance + confidence per item; let user correct; never trust mtime alone for owner-history insights (the copy itself rewrites mtime).
  - [medium] O(n^2) dedup / loading 1M hashes in memory exhausts RAM and stalls the app. -> Stream from SQLite, block by size+content-hash, index near-neighbor with BK-tree/LSH, bound worker queue, checkpoint progress for resumability; cap in-memory candidate windows.
  - [medium] RAW + sidecar + live-photo relationships get split across the dedup/organize passes, breaking the very 'integrity' the app promises. -> Treat RAW+JPEG, HEIC+MOV live pairs, and image+.xmp as atomic units before clustering; move/copy them together; never dedup a RAW against its derived JPEG.
  - [medium] ExifTool process-per-batch or ffmpeg spawns become a throughput bottleneck or leak processes during long 1M scans. -> Single long-lived ExifTool -stay_open cluster with bounded concurrency; pool + reuse, hard timeouts and kill-on-hang for ffmpeg/ffprobe spawns; backpressure via the worker queue.
  - [low] GPL ffmpeg build creates licensing exposure for a distributed desktop app. -> Ship an LGPL-compatible ffmpeg/ffprobe build, document provenance/version, isolate it as an external binary (not statically linked into app code).
MILESTONES:
  - M0 — Probe & catalog (MVP): Walk source tree incrementally; classify image/video; exiftool batch extract + ffprobe JSON probe into SQLite; resumable scan with (path,size,mtime) keying; basic format/resolution/codec stats. No hashing yet.
  - M1 — Exact dedup + thumbnails: xxhash content hash for exact dups; sharp thumbnails for UI; RAW+JPEG / live-photo / sidecar pairing; screenshot & scan classification; exact-dup report (non-destructive).
  - M2 — Perceptual dedup: dHash+pHash for images via worker pool; BK-tree near-neighbor; two-tier near-dup clustering with keeper selection; tunable thresholds + review UI.

## [ux] UX flows & human-in-the-loop review
SUMMARY: The UX is built around one killer loop: surface low-confidence album reconstructions and let the user confirm/correct them in seconds. The app is a 4-stage journey — Onboarding/Scan (live progress over 1M files), Review Workspace (the heart: confidence-sorted album candidates with side-by-side candidate-vs-current, "why grouped" explanations, drag-between-albums, bulk actions, manual MusicBrainz search), Insights (collection + owner-profile dashboards), and Organize (dry-run diff → confirm → copy execution). React + Vite renderer talks to the engine over a typed IPC bridge; all heavy work runs off the UI thread (engine process/sidecar), the UI only ever renders snapshots from a SQLite-backed read model, and every list is virtualized so 1M rows never hit the DOM. Responsiveness is a hard invariant: the renderer never blocks, never holds the full dataset, and re-renders only on diffed deltas.
RECO: Make the Review Workspace a keyboard-first triage queue, not a file browser. The user lands on the single highest-impact ambiguous album, sees candidate-vs-current side by side with a plain-language "why grouped" rationale, and resolves it with one keystroke (Enter=accept, R=reject, M=manual search, [/]=prev/next), then auto-advances. Treat review like a Gmail/Superhuman triage inbox sorted by confidence×size, with bulk "accept all above 95%" as the pressure-release valve. Everything else (insights, organize) is secondary to making this loop feel instant.
DECISIONS:
  - Component + chart library stack => Headless primitives (Radix UI / shadcn-style) for controls + TanStack Table (headless) for grids + ECharts for visualization, styled with Tailwind.
  - Virtualization strategy for 1M rows => TanStack Virtual windowing everywhere lists/trees appear; the renderer never receives more than a windowed page from the engine.
  - State management split => TanStack Query for engine/server-state (queries, pagination, cache, optimistic mutations) + zustand for ephemeral UI/edit/staging state. No Redux.
  - Keeping the UI responsive while the engine churns => Engine runs entirely off the renderer (separate process for Electron, Rust core + Node sidecar for Tauri); UI only renders read-model snapshots and receives throttled/batched delta events; long lists virtualized; expensive derived data computed in the engine, not the renderer.
  - Review interaction model => Keyboard-first triage queue with auto-advance, sorted by review-priority score, plus aggressive bulk actions.
  - File safety in the UI contract => Source is presented as immutable everywhere; the only write path is the Organize flow which requires an explicit dry-run review + Confirm before any copy.
ALGORITHMS:
  - Review-priority ranking: Order the triage queue by a score that surfaces high-impact ambiguity first: priority = (1 - confidence) weighted, boosted by track-count (bigger albums = bigger payoff) and by completeness gap (albums missing tracks vs the matched release), with conflicts (contradictory signals) floated to top. Confidence buckets: ≥0.95 auto-acceptable (one-click bulk), 0.75–0.95 quick-confirm, <0.75 full review. Score recomputed engine-side; UI just requests ordered pages.
  - 'Why grouped' evidence assembly: For each proposed group, the engine emits an evidence list: signal type (mb_release_id, acoustid_cluster, folder_colocation, album_artist_tag, track_number_continuity, year_match), the value, and a per-signal weight. UI renders supporting signals as positive chips (c-teal) and conflicting ones as warning/danger chips (c-amber/c-red), with overall confidence as the weighted aggregate. Expand to see raw matched values.
  - Optimistic edit + reconcile: User actions (accept, reject, move track, edit tag) apply immediately to the zustand staging buffer + TanStack Query optimistic cache, then dispatch to the engine; on engine confirm, reconcile the canonical read model; on failure, roll back the optimistic state and toast. Auto-advance to next queue item fires on accept/reject.
  - Delta batching / event coalescing: Engine emits progress and item-update events; the bridge coalesces them (progress throttled ~10/sec via requestAnimationFrame-aligned flush; item updates batched into arrays) before committing to the store, so React re-renders are bounded regardless of scan throughput.
  - Large-series chart down-sampling: For owner-profile scatter (up to ~1M points: file-mtime vs release-year) and histograms, request pre-aggregated/bucketed data from the engine (engine does the SQL GROUP BY); for raw scatter use ECharts large-mode + LTTB sampling so the chart stays interactive.

# ============ SYNTHESIS ============

## productSummary
"sabiorg" is a cross-platform desktop application (all-TypeScript) that rescues a scattered personal media library by RECONSTRUCTING album integrity for music first, then the analogous units for photos (events) and video (series/home-video events). It scans hundreds of thousands to 1M+ files non-destructively (source is always read-only), reconstructs releases via tag/folder/fingerprint entity resolution with hybrid online enrichment (AcoustID + MusicBrainz + Cover Art Archive + Discogs) and graceful offline fallback, lets the user review/correct groupings in a fast keyboard-first triage UI, edits metadata, and COPIES corrected, well-tagged files into a clean destination tree (Artist/Album/NN - Track). It then surfaces two classes of insight: about the COLLECTION (genres, format/lossless mix, bitrate/quality, duplicates, completeness, fake-FLAC) and about the OWNER (acquisition cadence from build-history timestamps, classic-vs-new ratio, taste/archetype profiling). The core promise: never mutate originals, scale to 1M items with bounded memory and resumable scans, and make the album-review loop feel instant."

## architectureOverview
"Single language (TypeScript) end to end on Electron 33+, chosen over Tauri-v2 because every heavy workload (better-sqlite3 bulk writes, sharp/libvips, worker_threads, fpcalc/ffprobe subprocesses, MusicBrainz HTTP) is Node-native and IO/subprocess-bound — a Tauri build would still ship a Node sidecar, paying Rust+Node packaging cost for no compute win. Three-tier process model: (1) a thin React renderer with ZERO Node access (contextIsolation on, sandbox on, nodeIntegration off) that only renders windowed snapshots from a SQLite-backed read model; (2) a coordinator main process for lifecycle/windows/menus/auto-update/safeStorage and typed IPC routing; (3) a long-lived utilityProcess 'engine host' (own PID, survives renderer reload) that owns the SQLite connection, the resumable job scheduler, and drives a Piscina worker_threads pool (CPU work: tag parse, hashing, perceptual hashing, FFT) plus per-binary execa subprocess pools (fpcalc, ffprobe/ffmpeg). The domain logic lives in a pure, Electron-free @sabiorg/engine library reused verbatim by a headless @sabiorg/cli, enforced by a dependency-cruiser import boundary in CI. The ingestion pipeline is a cheap-first staged state machine (discover -> stat -> probe/tag -> content-hash -> fingerprint -> enrich -> reconstruct -> organize) where SQLite IS the durable work queue (one row per file, per-stage status columns), giving free crash recovery, idempotent resume, and delta re-scans. Reconstruction inverts naive matching into constrained set-cover: pick the MusicBrainz RELEASE best COVERED by a candidate cluster (after multi-key blocking + union-find), yielding correct album cohesion and completeness for free. Storage is a single embedded SQLite (better-sqlite3, WAL, FTS5) with Drizzle for typed schema/migrations; every derived metadata value is an append-only provenance-stamped MetadataClaim collapsed into a materialized ResolvedField projection so provenance, re-enrichment, offline fallback, and human overrides all fall out naturally. Heavy analytics run in an attached DuckDB read layer over incrementally-maintained rollup tables so 1M-row dashboards stay interactive without contending with scan writes. The only safe write path is the Organize flow: copy bytes -> verify copy hash -> tag the COPY (never the source) -> journal every action idempotently for resume."

# ============ REVIEWS ============

## LENS: Completeness & correctness — adversarial review of the album-reconstruction algorithm, edge cases, and missing/under-specified components
VERDICT: The plan is unusually thorough on infrastructure (process model, queue-as-SQLite, provenance claims, copy-then-verify safety) and the headline set-cover inversion is genuinely the right call. But the reconstruction algorithm has real soundness gaps that will surface as user-visible wrong groupings, and several first-order capabilities are missing or hand-waved: CUE/single-file album images, multi-disc set-cover, the cold-start chicken-and-egg between blocking and fingerprinting, MusicBrainz replica operational reality, and the labeled-corpus that the entire precision story rests on. The blocking strategy as written will both over-merge and under-recall in common real-world layouts. This is a strong plan that should not be approved as "sound" without closing the reconstruction correctness items below.
FINDINGS:
  - [high] The set-cover release disambiguator is specified as a single-medium (single-disc) algorithm but the data model and edge-case notes assume multi-disc. 'coverage = #positions of Rel filled / totalTracks(Rel)' has no medium/disc dimension. A correct multi-disc release has tracks numbered 1..N PER MEDIUM, so naive position assignment will collide (two 'position 1' tracks), and a cluster that is actually CD2 of a 2-disc set will score catastrophically low coverage against the full release and be rejected or split. The Hungarian/greedy assignment described operates over a flat position space. Multi-disc is the single most common 'broken album integrity' case and the core algorithm does not actually handle it.
      FIX: Make the cover/assignment operate over (mediumPosition, trackPosition) tuples. Score coverage per-medium and allow a cluster to match a SUBSET of media (a cluster may legitimately be only one disc of a box set). Add explicit 'partial-medium' candidate states so CD2-only folders resolve to 'Release X, Disc 2, complete' rather than 'Release X 50% incomplete'. Re-merge sibling disc folders via shared releaseGroup/barcode BEFORE final scoring.
  - [high] Chicken-and-egg between blocking and the strong keys it depends on. Blocking treats AcoustID id (K4) and tagged releaseGroup MBID (K3) as the only single-key merges, but K4 requires fingerprinting+network resolution, which the ingestion facet explicitly GATES behind 'album reconstruction confidence below threshold'. So the cheap offline blocking pass cannot use the one high-precision signal, and the network pass is gated on the very grouping it would improve. For untagged/scattered libraries (the stated core problem) K3 is absent and K4 is unavailable at blocking time, leaving only the weak folder/album-tag keys — exactly the signals that are broken in the target collections.
      FIX: Make this an explicit two-phase loop: (1) offline blocking on weak keys produces provisional clusters; (2) fingerprint only the AMBIGUOUS/low-cohesion clusters; (3) RE-BLOCK using the new AcoustID/recording keys and merge/split. The plan implies one-shot blocking. Specify the convergence/iteration and how cluster identity is preserved across re-blocking so review decisions aren't invalidated.
  - [high] CUE + single-file album images (one FLAC/APE/WAV per album with an embedded or sidecar .cue) are mentioned once in the reconstruction facet as an edge case ('detect .cue, split logical tracks before featurization') and then dropped entirely from the data model, organize engine, and roadmap. This is a pervasive real-world format for ripped albums and is itself a perfectly-intact album that the engine would mis-handle: one MediaFile = N logical tracks. There is no representation for virtual/logical tracks, no organize-time decision (split into per-track files vs copy image+cue), and TrackFile assumes 1 file = 1 track.
      FIX: Either declare CUE-split explicitly out of scope for v1 (and detect+quarantine such files so they aren't silently mistagged) or model logical tracks (a TrackFile that references a parent image file + start/end offsets) end-to-end. The current 'split before featurization' note implies transcoding/decoding on the read path, which contradicts the copy-only, read-only-source guarantee — splitting produces NEW audio, a major scope item that is unbudgeted.
  - [high] Blocking over-merge risk from key K2 = norm(albumArtist)+'|'+norm(album) combined with the 'two of {K1,K2,K5} corroborate' rule. K2 and K5 are near-duplicates (both contain norm(album)); a track with a populated album tag and a year will frequently satisfy BOTH K2 and K5 by itself, meaning a SINGLE underlying signal (the album tag) masquerades as 'two agreeing strong keys' and triggers a merge. This defeats the stated mitigation 'require >=2 agreeing strong keys.' Greatest-hits/self-titled/'Live' albums across different artists, and the classic 'Track 01'/'Unknown Album' sentinels, will collapse together.
      FIX: Treat album-derived keys as ONE signal source, not two. Require corroboration from INDEPENDENT signal families (folder cohesion, fingerprint cluster, embedded cover hash, MBID) not two views of the same tag. Expand the sentinel/exclusion list and make it data-driven (any album value shared by >K distinct albumArtists is auto-demoted to a non-merge key).
  - [medium] The 'classic-vs-new' owner profiling and acquisition-cadence insights rest on filesystem timestamps that the plan ITSELF acknowledges are unreliable, AND the plan's flagship copy-only organize step resets them. The mitigation is 'snapshot acquisition time at scan before organize.' But there is no guarantee the user scans BEFORE a prior copy/sync already destroyed the original timestamps — for a 'scattered, downloaded, re-copied' collection (the stated target), source btime/mtime are ALREADY fiction at first scan. The profiling feature may produce confident-looking output that is fundamentally noise for the exact collections the product targets.
      FIX: Be honest in the design: gate the entire owner-timeline behind a reliability detector (already partly specified) and default it OFF / 'insufficient reliable history' for collections showing timestamp-collapse. Consider mining acquisition signals that survive copying (purchase-date tags, iTunes 'date added' atoms, download-folder structure, filename date stamps) as primary rather than fs times. Right now fs-time is primary and the survivable signals are an afterthought.
  - [high] The MusicBrainz local Postgres replica is presented as the load-bearing solution to the rate-limit problem ('primary bulk source above ~50k items') but is wildly under-budgeted for a DESKTOP app. The full MB replica is a multi-hundred-GB Postgres database requiring mbslave/replication packet ingestion, ongoing Live Data Feed sync, and a running Postgres server — none of which fit the 'embedded, offline-first, single-file-backup, modest-machine' framing elsewhere in the same plan. The roadmap (M4) lists 'LOCAL MusicBrainz Postgres replica backend' as a normal milestone item with no provisioning UX, disk preflight, or sync-maintenance design. This is the plan's biggest scope/feasibility gap hiding behind a one-liner.
      FIX: Resolve the contradiction explicitly. The 'fallback to content-hash-on-suspicion' is mentioned for change detection but NOT wired into move/decision-preservation, which is keyed on (dev,ino).

## LENS: Scale & feasibility at 1M+ items in a desktop app (memory, DB throughput, fingerprinting/DSP throughput, online rate limits, TB-scale copy)
VERDICT: The architecture is internally coherent and the team clearly knows the standard playbook (SQLite-as-queue, streaming, virtualization, set-cover matching, local MB replica). But the plan systematically conflates "we have a bounded-memory design" with "we have demonstrated 1M throughput," and several headline numbers are wishful. The two genuinely load-bearing scale claims — "cold walk + tag-read of 1M files in single-digit minutes" and "DuckDB keeps 1M-row dashboards interactive" — are asserted, not derived, and the first is almost certainly false on real consumer hardware/HDDs. The hardest physical realities (decoding 1M audio files for fingerprinting, copying TBs through a verify+tag+re-read pipeline, the irreducible AcoustID bottleneck, and the operational burden of a local MusicBrainz Postgres replica on a "desktop app for a normal user") are acknowledged as risks but their consequences are not propagated into the roadmap, the exit criteria, or the product promise. The plan also quietly assumes a power-user with a local Postgres replica as the ONLY tractable path, which contradicts the "personal media librarian" framing. Net: feasible as an engineering platform, but the 1M+ "interactive desktop app" promise is oversold and the milestone time estimates ignore the dominant cost (I/O and decode wall-clock, not code).
FINDINGS:
  - [high] The headline throughput claim is unsubstantiated and likely false on real hardware. M1 exit criteria states 'Cold walk + tag-read of a synthetic 1M-file fixture in single-digit minutes with bounded heap.' Single-digit minutes for 1M files means >1,600–3,300 files/sec sustained, EACH requiring an open + partial streaming read (music-metadata parseStream reads header/tag region) + a SQLite write. On a SATA SSD with ~100µs random open latency that is plausible only with massive parallelism and a warm OS cache; on a 5400rpm HDD (where large scattered media hoards actually live) random small reads cap at a few hundred IOPS, putting cold scan in the HOURS, not minutes. The 'synthetic 1M-file fixture' will mask this because synthetic files are small, contiguous, and freshly written (warm cache, no fragmentation) — it validates the code path but not the real-world wall-clock. The plan benchmarks against the wrong fixture.
      FIX: Drop the 'single-digit minutes' promise or qualify it to 'SSD, warm cache.' Add a fixture that simulates HDD seek latency and real file-size distribution (FLAC 20-40MB, scattered dirs). Make M1 exit criteria a files/sec throughput floor measured on a spinning disk, and surface a realistic ETA in the UI rather than promising a fixed time.
  - [high] The 1M+ audio fingerprinting decode cost is grossly under-weighted and the 'need-predicate gating' is doing all the load-bearing work without justification. fpcalc must DECODE each audio file (it bundles ffmpeg) to compute the Chromaprint — that is full-file decode, not a header read. At even 0.3–1s of CPU per track decode, 1M tracks = 80–280 CPU-hours, i.e. many hours to a day+ wall-clock even across 8 cores, plus disk read of the entire audio payload (TBs). The plan repeatedly hand-waves this with 'only ambiguous/unidentified files reach fpcalc,' but offers NO estimate of what fraction that is. For the user's stated problem — a scattered, MISTAGGED, UNTAGGED hoard — the ambiguous fraction is plausibly the MAJORITY of files, which is exactly the population that defeats the gate. The whole 'enrichment is a cheap background upgrade' story collapses if 60% of a messy library needs fingerprinting.
      FIX: Estimate (and instrument) the expected fingerprint-eligible fraction for a genuinely messy library, not a clean one. Treat full-library fingerprinting as a multi-hour-to-overnight batch with explicit progress/ETA, not a 'background upgrade.' Consider partial-file fingerprinting (fpcalc -length to limit decode duration) and aggressive pre-fingerprint audio-content dedup as first-class, not footnotes. Re-baseline the M4 duration estimate against decode wall-clock.
  - [high] The local MusicBrainz Postgres replica — the linchpin that makes 1M enrichment 'tractable' — is fundamentally incompatible with a self-contained desktop app for a normal user, and the plan knows this but proceeds anyway. The entire enrichment scale story rests on 'make a LOCAL MusicBrainz replica the primary bulk source above ~50k items' because live API is ~11.6 days. But a MB replica means: a full Postgres install, ~50GB+ of data import, mbslave/Live Data Feed replication setup, tens of GB of disk, and ongoing ops — for a 'personal media librarian.' This is a server-admin task, not a desktop install. So for the actual target persona at 1M scale, there is effectively NO viable online-enrichment path: live API is too slow, and the replica is too heavy. The plan resolves a conflict on paper ('source-agnostic resolver, two backends') while leaving the real user stranded.
      FIX: Be honest that at true 1M scale without a replica, online enrichment is partial/best-effort, and design the offline+tag+heuristic path to be genuinely good (it is the real default at scale), not a degraded fallback. Investigate shipping a prebuilt, read-only, embedded MB slice (e.g. a curated SQLite/Parquet export of releases+recordings, refreshed via app updates) rather than asking users to run Postgres. State the disk-cost honestly in product copy.
  - [medium] DuckDB 'interactive 1M-row dashboards' adds a SECOND heavyweight native engine and a dual-write/sync problem, and the interactivity claim is asserted without accounting for rollup maintenance cost. The plan runs SQLite (OLTP + queue) AND DuckDB (OLAP) with 'incrementally-maintained rollup tables' driven by a dirty-row set. This doubles the native-dependency packaging risk (already flagged as the #1 risk) and introduces rollup drift (also flagged) — to solve a problem (slow GROUP BY) that well-indexed SQLite + precomputed summary tables already largely solves at 1M rows. DuckDB's win is real for ad-hoc OLAP, but the plan's own rollup strategy means dashboards read from small precomputed tables anyway, so DuckDB's columnar speed is mostly redundant with the rollups. You pay two native engines for marginal benefit, and the 'reconcile rollups against base tables on startup' check is an O(N) full scan that itself fights the scale budget.
      FIX: Defer DuckDB (the plan already says M5) but go further: prove that SQLite + materialized rollup tables can't hit the dashboard latency target BEFORE adopting DuckDB at all. If rollups are maintained, you may never need the columnar engine. If you keep DuckDB, use sqlite_scanner to avoid a second copy of the data, and make the startup reconciliation sampled/incremental, not a full O(N) scan.
  - [medium] TB-scale copy with copy → verify-rehash → tag-the-copy → read-back-assert is a 3-4x read/write amplification over the dataset that the time estimates ignore. The organize pipeline (M3) does: copy bytes, re-hash the COPY (full read of every copied byte) for blake3 verify, write tags, then optionally re-hash/re-read for final assertion. For a multi-TB music library that is: write all bytes once + read all bytes back at least once (verify) + re-read for tag assert. On a single consumer disk at ~150MB/s effective, multiple TB of read+write amplification is many hours of pure I/O, and if source and dest share one spindle, seeks make it dramatically worse. M3's '5-6 weeks' is dev effort but the SOAK RUN exit criterion ('large-corpus soak run completes') will take a full day+ of wall-clock per run, which will throttle iteration. blake3 verify on every file is defensible but doubles read I/O.
      FIX: Quantify expected copy wall-clock at target TB sizes and same-disk vs cross-disk scenarios; surface it in the dry-run estimate. Reconsider full blake3 verify on EVERY file vs verifying via the copy's own post-write hash without a separate re-read pass (compute hash streaming DURING copy, not in a second read). Make the read-back tag assertion sampled by default, not per-file, to cut a full extra read pass.
  - [medium] AcoustID at 3 req/s is the one irreducible online bottleneck and the plan admits it (~3.9 days for 1M unique) but then folds it into 'background, resumable' as if duration were a non-issue. Even with perfect dedup halving the set, that's ~2 days of continuous network at the rate cap, assuming zero failures, zero retries, and the user leaving the app running. The plan never asks whether a personal-librarian user will tolerate a multi-day background job, nor what the experience is across app restarts, sleep/hibernate, network changes, and laptop usage patterns. A 2-4 day job on a desktop that sleeps when the lid closes may NEVER complete in practice.
      FIX: Design for the realistic case that AcoustID enrichment is never fully completed for a 1M library. Prioritize fingerprinting the files that most need it (low tag confidence, ambiguous clusters) so the first hours deliver the most value, and make the product useful at any % of enrichment. Handle sleep/wake and network-loss explicitly in the limiter/resume logic. Set user expectations that full fingerprinting is opt-in and long-running.
  - [medium] Union-find blocking with 'mega-block caps' is a correctness/scale tension that is acknowledged but the cap creates silent recall loss at exactly the scale where it matters. The plan caps block size (~400) and splits oversized blocks. For a messy hoard, generic folders ('Music', 'Downloads', 'Unknown Album') and blank album tags create huge blocks; capping/splitting them means true co-album members can land in different sub-blocks and never be compared, scattering albums — the precise failure the set-cover inversion was meant to fix. At 1M with poor tags, a large fraction of files may have weak/sentinel keys and thus get degraded clustering. The 'precision/recall measured on a labeled corpus' exit criterion will look fine on a CLEAN labeled corpus and hide this.
      FIX: Build the labeled reconstruction corpus from genuinely MESSY data (untagged, sentinel-tagged, split-across-folders), not clean releases, so recall loss from block-capping is visible. Consider fingerprint-cluster (AcoustID) as a strong cross-block merge key specifically to recover albums that weak tag/folder keys scatter — but note this re-introduces the fingerprint-cost problem above, making the dependency explicit.
  - [medium] Bounded-memory is asserted across many independent in-memory structures whose combined footprint at 1M is never summed. union-find over 1M tracks, BK-tree/LSH over 1M perceptual/audio hashes held for near-dup queries, the dirty-row set, per-stage queues, TanStack Query cache, plus better-sqlite3's own page cache and mmap. Each is individually 'streamed' or 'bounded,' but a BK-tree over 1M 64-bit hashes plus node overhead, kept resident for radius queries, is easily hundreds of MB to GB in JS; union-find parent arrays for 1M+ are tens of MB; SQLite mmap ('large mmap' per datamodel notes) can map gigabytes. The plan has a 'heap watchdog' but no memory budget. On a 4-8GB consumer laptop running Electron (Chromium already ~hundreds of MB), this can OOM.
      FIX: Produce an explicit per-component memory budget at 1M (and 5M) items and a total ceiling for a minimum-spec machine (e.g. 8GB). Store BK-tree/LSH in SQLite (sqlite-vec or on-disk structure) rather than resident JS objects, or build it per-blocking-partition rather than globally. Cap SQLite mmap/page cache explicitly.   - [medium] Native-dependency matrix is enormous and is the self-identified #1 risk, yet the plan keeps ADDING native engines, undercutting its own mitigation. The bundled/native surface: better-sqlite3, sharp/libvips, DuckDB, fpcalc(+ffmpeg), ffprobe/ffmpeg-static, exiftool(+Perl), @node-rs/xxhash, plus optional pg and sqlite-vec — across Win/mac/Linux × x64/arm64 × every Electron ABI bump. exiftool-vendored alone drags a Perl runtime. The plan's mitigation (CI matrix + packaged smoke test) is correct but the plan simultaneously chose to maximize the number of things that matrix must cover, including a second DB (DuckDB) whose value is contested above.
      FIX: Aggressively minimize the native surface: prefer WASM where viable (hash-wasm for ALL hashing to drop @node-rs/xxhash; blockhash-core fallback already noted), defer/drop DuckDB, and gate exiftool to the image/video milestone only. Treat each native dep as a recurring tax in every Electron upgrade and weigh it against a pure-WASM/pure-TS alternative explicitly in the stack table.
  - [low] The roadmap durations (totaling ~45-54 weeks across M0-M7) measure engineering effort but ignore that scale validation is gated on long-running soak jobs, making the real calendar far longer. Several exit criteria ('1M fixture candidates generate,' 'large-corpus soak run completes,' '1M+ soak run is stable, bounded-memory, resumable,' 'reconstruction precision/recall on the labeled corpus') each require generating and processing a 1M+ corpus — potentially TBs of synthetic audio for realistic fingerprint/copy testing. The estimates read as if these are fast unit-test gates.
      FIX: Add an explicit M0/M1 line item for building a TIERED test corpus (small clean, large clean, large messy, TB-scale realistic-audio) and budget the soak-test wall-clock into milestone calendars. Recognize that scale exit criteria gate the schedule by job duration, not code, and parallelize soak runs in CI infra.
  - [low] Owner-profiling from filesystem timestamps is acknowledged as fragile but is still promised as a headline insight, and at 1M scale the 'single import day' collapse is the COMMON case, not an edge case. A genuinely scattered hoard that someone is finally organizing was very likely bulk-copied/restored from old drives/cloud, which flattens birthtime/mtime to import dates. The feature is sold as core but is structurally likely to be unavailable for the core persona.
      FIX: Demote build-history/acquisition-cadence from a headline promise to a 'when timestamps are reliable' bonus. ## LENS: Sequencing & MVP realism — is the roadmap ordered to deliver core music-album value fastest, what is over-engineered for MVP, and what is the smallest lovable v0?
VERDICT: The roadmap is intellectually coherent but badly front-loads infrastructure and back-loads the actual user payoff. The plan's own "core promise" is rescuing scattered albums, yet a user gets ZERO album value until the end of M2 — which by the plan's own estimates is week 15–18 (M0 3–4w + M1 5–6w + M2 7–8w). That is a quarter-plus before the first lovable moment. Worse, the single most painful, most demo-able outcome — "here is a corrected, well-tagged copy of an album on disk" — does not land until the END of M3 (~week 20–24). The sequencing optimizes for architectural purity (M0 spends 3–4 weeks on monorepo boundaries, dependency-cruiser CI, 6-OS packaging smoke tests, credit-based IPC) before a single album is ever reconstructed. For a greenfield personal-librarian tool whose value is unproven, this is the classic mistake of building the cathedral before checking anyone wants to pray in it. The plan also conflates "scale to 1M" with "MVP": M1's exit criteria demand a synthetic 1M-file fixture passing in single-digit minutes with bounded heap BEFORE any reconstruction exists. You are hardening throughput for a feature you haven't validated. The genuinely correct insight buried in the plan — that offline reconstruction (M2) precedes online enrichment (M4) — is right and should be protected. But the smallest lovable v0 is roughly "M1-lite + M2-lite + M3-lite on a few thousand files," and the plan offers no such vertical slice; every milestone is a horizontal infrastructure layer. Defer DuckDB, set-cover-over-a-local-MusicBrainz-replica, the provenance claim/ResolvedField machinery, photos/video, and most of M7 until after a human has actually used the album loop on their real library.
